> **阶段目标：** 进程创建时向内存管理器申请内存，进程终止时自动归还，终端能打印出可视化的内存布局图。

## 实验一：理解内存分区，定义数据结构

### 目标
定义内存块（`MemBlock`）结构体，初始化一块全空闲的内存，并能打印内存布局。

### 背景知识：动态分区分配

操作系统把内存想象成一条连续的地址空间（比如 0 到 1023，代表 1024KB）。

进程来了就从里面划出一块给它用，进程走了就把那块标记回"空闲"。这就是**动态分区分配**。

内存在任意时刻，都是若干已分配块和空闲块交替排列的链表：

```
地址：  0      200    350    650    800    1024
        ├──P1──┤─FREE─┤──P2──┤─FREE─┤──P3──┤
```

每一块需要记录：

| 字段 | 类型 | 含义 |
|------|------|------|
| `start` | `int` | 块的起始地址（KB） |
| `size` | `int` | 块的大小（KB） |
| `pid` | `int` | 占用该块的进程 PID（-1 表示空闲） |
| `process_name` | `std::string` | 进程名（空闲块为 "FREE"） |

用 `std::list<MemBlock>` 存储所有内存块，而不是 `vector`。原因：分配和释放时需要在中间插入/删除节点，`list` 的时间复杂度是 O(1)，`vector` 是 O(n)。

### 步骤

**1. 新建 `src/memory/memory_manager.h`**

需要定义：
```
struct MemBlock {
    // 上表中所有字段
};

class MemoryManager {
public:
    // 构造函数：接受总内存大小（KB），初始化为一整块空闲内存
    MemoryManager(int total_kb);

    // 打印当前内存布局（ASCII 可视化）
    void printLayout() const;

    // 统计并打印使用率
    void printStats() const;

private:
    int total_size;
    std::list<MemBlock> blocks;   // 内存块链表，按地址顺序排列
};
```

**2. 新建 `src/memory/memory_manager.cpp`**

实现构造函数：初始化时，`blocks` 里只有一个元素：
```
MemBlock { start=0, size=total_size, pid=-1, process_name="FREE" }
```

实现 `printLayout`，打印类似下面的效果：

```
=== 内存布局（1024KB）===
[   0 -  199] 200KB  P1(init)
[ 200 -  349] 150KB  FREE
[ 350 -  649] 300KB  P2(gcc)
[ 650 - 1023] 374KB  FREE
```

格式自由，但每块的起止地址、大小、占用状态必须显示。

实现 `printStats`：
```
已用：500KB / 1024KB（48.8%）  空闲：524KB
```

**3. 修改 `CMakeLists.txt`**

在 `add_executable` 的文件列表中追加 `src/memory/memory_manager.cpp`。

**4. 修改 `src/main.cpp`**，创建 `MemoryManager` 并打印初始状态

```cpp
#include "memory/memory_manager.h"

// 在 main() 里，进程列表创建后加：
MemoryManager mem(1024);   // 1024KB 物理内存
mem.printLayout();
mem.printStats();
```

### 检查点
- [ ] `MemoryManager` 构造后，`printLayout` 能显示"一整块 1024KB FREE"
- [ ] `printStats` 显示 0% 使用率
- [ ] 编译通过，`./minios` 输出内存初始状态

## 实验二：实现首次适应算法，为进程分配内存

### 目标
实现 `firstFit` 分配函数，进程创建（`NEW → READY`）时分配内存，并更新进程的 `memory_base` 字段。

### 背景知识：首次适应（First Fit）

从内存链表头部开始扫描，找到**第一个**大小 ≥ 请求大小的空闲块，将其分割：

```
申请 200KB 前：
[ 0 - 1023]: 1024KB FREE

申请 200KB 后：
[ 0 -  199]: 200KB  P1(init)   ← 分配给进程
[200 - 1023]: 824KB FREE        ← 剩余空闲块（原块被分割）
```

如果剩余部分 > 0，需要在链表里插入一个新的 FREE 块，不能丢掉。

### 步骤

**1. 在 `memory_manager.h` 中新增 `allocate` 函数声明**

```cpp
// 为进程 pid 分配 size KB 的内存
// 成功：返回分配到的起始地址（填入 PCB 的 memory_base）
// 失败：返回 -1（内存不足）
int allocate(int pid, const std::string& name, int size);
```

**2. 在 `memory_manager.cpp` 中实现 `allocate`（首次适应）**

算法逻辑：
```
遍历 blocks 链表，找第一个满足：pid == -1 且 size >= 请求大小 的块

找到后：
  1. 记录该块的起始地址 addr = 该块的 start
  2. 计算剩余大小 remaining = 该块 size - 请求大小
  3. 将该块改为已分配：pid = 进程pid，name = 进程名，size = 请求大小
  4. 如果 remaining > 0，在该块之后插入一个新的 FREE 块：
       start = addr + 请求大小，size = remaining，pid = -1
  5. 返回 addr

没找到：返回 -1，打印 "内存不足，进程 xxx 分配失败"
```

**3. 在 `src/process/process.h` 的 `admit()` 函数中，接入内存分配**

修改 `admit()` 的签名，让它接受一个 `MemoryManager` 引用：

```cpp
void admit(MemoryManager& mem);
```

在 `admit()` 的实现中，状态切换之前先申请内存：
```
int addr = mem.allocate(pid, name, memory_size);
if (addr == -1) {
    // 内存不足，进程无法就绪，打印提示
    return;
}
memory_base = addr;
state = ProcessState::READY;
// 打印日志：[PID=1] init: NEW → READY，内存: 0KB ~ 100KB
```

**4. 修改 `main.cpp`，将 MemoryManager 传给 admit()**

```cpp
for (auto& p : processes) {
    p.admit(mem);
}
mem.printLayout();
```

### 预期输出

```
=== 内存分配 ===
[PID=1] init:   NEW → READY，内存分配: 0 ~ 99KB
[PID=2] shell:  NEW → READY，内存分配: 100 ~ 299KB
[PID=3] vim:    NEW → READY，内存分配: 300 ~ 449KB
[PID=4] gcc:    NEW → READY，内存分配: 450 ~ 749KB
[PID=5] python: NEW → READY，内存分配: 750 ~ 829KB

=== 内存布局 ===
[  0 -   99]  100KB  P1(init)
[100 -  299]  200KB  P2(shell)
[300 -  449]  150KB  P3(vim)
[450 -  749]  300KB  P4(gcc)
[750 -  829]   80KB  P5(python)
[830 - 1023]  194KB  FREE

已用：830KB / 1024KB（81.1%）  空闲：194KB
```

### 检查点
- [ ] 5个进程全部分配到内存，`memory_base` 不再是 -1
- [ ] 内存布局图正确，总大小等于 1024KB
- [ ] 申请内存后 admit() 里的状态切换日志格式包含内存地址信息

## 实验三：进程终止时回收内存，合并空闲块

### 目标
实现 `free` 函数，进程终止时释放内存；相邻的空闲块自动合并，避免碎片积累。

### 背景知识：外部碎片与合并

进程释放内存后，可能出现多个零散的小空闲块，总量足够但每块都太小，导致大进程无法分配——这就是**外部碎片**。

解决方法：每次释放后，检查并合并相邻的空闲块：

```
释放 P2(shell) 之前：
[  0-99]  P1   [100-299]  P2   [300-449]  P3   [450-749]  P4   [750-829]  P5   [830-1023]  FREE

释放 P2(shell) 之后（合并前）：
[  0-99]  P1   [100-299]  FREE   [300-449]  P3   ...

如果再释放 P1(init)：
[  0-99]  FREE   [100-299]  FREE   [300-449]  P3   ...
↓ 合并相邻空闲块
[  0-299]  FREE   [300-449]  P3   ...   （两个相邻 FREE 合并成一个）
```

### 步骤

**1. 在 `memory_manager.h` 中新增两个函数**

```cpp
// 释放进程 pid 占用的内存块，并合并相邻空闲块
void free(int pid);

// 合并相邻的空闲块（由 free 内部调用，也可以单独暴露出来测试）
void mergeAdjacentFree();
```

**2. 实现 `free`**

```
遍历 blocks，找到 pid == 该进程 PID 的块
将其标记为空闲：pid = -1，process_name = "FREE"
调用 mergeAdjacentFree()
```

**3. 实现 `mergeAdjacentFree`**

用迭代器遍历链表，当当前块和下一块都是空闲时，合并：
```
auto it = blocks.begin();
while (it != blocks.end()) {
    auto next = std::next(it);
    if (next == blocks.end()) break;
    if (it->pid == -1 && next->pid == -1) {
        // 合并：把 next 的大小加到 it，删除 next
        it->size += next->size;
        blocks.erase(next);
        // 不要 ++it，继续检查当前块和新的下一块
    } else {
        ++it;
    }
}
```

**4. 修改 `process.h` 中的 `terminate()` 函数**

类似 `admit()`，让 `terminate()` 接受 `MemoryManager` 引用，在状态切换前归还内存：

```cpp
void terminate(MemoryManager& mem);
// 实现：mem.free(pid)，然后 state = TERMINATED
// 打印日志：[PID=1] init: RUNNING → TERMINATED，释放内存 0 ~ 99KB
```

**5. 在 `main.cpp` 中演示内存回收**

```cpp
// 先让所有进程就绪
for (auto& p : processes) p.admit(mem);
mem.printLayout();

// 模拟：P1 和 P3 执行完毕
processes[0].dispatch();
processes[0].terminate(mem);   // P1 终止，释放 100KB

processes[2].dispatch();
processes[2].terminate(mem);   // P3 终止，释放 150KB

mem.printLayout();             // 看看释放后内存布局
mem.printStats();

// 尝试分配一个新进程（此时有碎片）
MemBlock* result = // ...
// 再次打印，观察首次适应的行为
```

### 预期输出（P1、P3 释放后）

```
=== P1、P3 终止，内存回收 ===
[PID=1] init: RUNNING → TERMINATED，释放内存 0 ~ 99KB
[PID=3] vim:  RUNNING → TERMINATED，释放内存 300 ~ 449KB

=== 内存布局（回收后）===
[  0 -   99]  100KB  FREE
[100 -  299]  200KB  P2(shell)
[300 -  449]  150KB  FREE        ← 两个 FREE 没有相邻，不会合并
[450 -  749]  300KB  P4(gcc)
[750 -  829]   80KB  P5(python)
[830 - 1023]  194KB  FREE

已用：580KB / 1024KB（56.6%）  空闲：444KB（3块碎片）
```

### 检查点
- [ ] `free()` 正确将目标块标记为空闲
- [ ] `mergeAdjacentFree()` 在释放 P1 后，如果旁边有空闲块就合并
- [ ] `terminate()` 传入 MemoryManager 后，内存和状态都更新正确
- [ ] 打印碎片信息（空闲块数量）

## 实验四：对比最佳适应算法

### 目标
实现 `bestFit` 分配函数，与 `firstFit` 对比，观察哪种算法产生的碎片更少。

### 背景知识：最佳适应（Best Fit）

不是找**第一个**够大的块，而是找所有够大的块中**最小的那个**：

```
空闲块：[100KB FREE]  [400KB FREE]  [200KB FREE]
申请 150KB：

首次适应：选 100KB？不够。选 400KB → 分配后剩 250KB FREE
最佳适应：找最小的够大的 → 选 200KB → 分配后剩 50KB FREE
```

最佳适应留下的碎片更小，但搜索需要遍历整个链表。

### 步骤

**1. 在 `memory_manager.h` 中新增 `bestFit` 声明**

```cpp
int bestFit(int pid, const std::string& name, int size);
```

**2. 实现 `bestFit`**

算法逻辑（与 `firstFit` 的区别只在查找阶段）：
```
遍历所有空闲块，记录满足条件（pid==-1 且 size >= 请求）的块中，size 最小的那个
找到后，分割逻辑与 firstFit 完全相同
```

**3. 在 `MemoryManager` 中添加一个策略选项**

在构造函数中增加参数，或者提供两个单独的接口，让 `main.cpp` 能选择用哪种策略。

**4. 在 `main.cpp` 中对比两种算法**

用相同的进程序列，分别用 `firstFit` 和 `bestFit` 分配，打印各自的布局和碎片情况，对比。

### 检查点
- [ ] `bestFit` 在同一进程序列下，选择的块与 `firstFit` 不同（可手算验证）
- [ ] 两种算法都能正确分割内存块
- [ ] 对比输出能看出碎片数量的差异

## 常见问题

**Q：`std::list` 的迭代器怎么用？和 `vector` 有什么区别？**  
A：`vector` 的迭代器支持随机访问（`it + 5`），`list` 的迭代器只支持前后移动（`++it`、`--it`）。`std::next(it)` 返回下一个迭代器，`blocks.erase(it)` 删除该节点并返回下一个迭代器的位置。

**Q：`list` 的 `insert` 怎么用？**  
A：`blocks.insert(pos, value)` 在 `pos` 之前插入 `value`。例如要在 `it` 后面插入，用 `blocks.insert(std::next(it), newBlock)`。

**Q：进程 `memory_base` 是 -1 时调用 terminate() 会怎样？**  
A：应该在 `terminate()` 里检查 `memory_base != -1` 才调用 `mem.free()`，否则会传入 -1 导致释放错误的块。加一个 `if` 判断即可。
