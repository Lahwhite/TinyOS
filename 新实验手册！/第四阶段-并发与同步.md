## 实验一：观察竞争条件，理解并发的危险

### 目标
写一个不加锁的多线程程序，让两个线程同时修改共享变量，观察竞争条件导致的错误结果。

### 背景知识：为什么并发会出错？

前三个阶段的调度器是**单线程模拟**：一个 `for` 循环按顺序处理进程，根本没有真正的并发。

真实操作系统里，多个进程/线程同时运行，共享内存。问题来了：

```
共享变量 counter = 0

线程1：counter++    实际上是三步：读 → 加 → 写
线程2：counter++    实际上是三步：读 → 加 → 写

若两个线程的三步交错执行：
  线程1 读到 counter=0
  线程2 读到 counter=0   ← 在线程1写回之前读，所以也读到 0！
  线程1 写回 counter=1
  线程2 写回 counter=1   ← 覆盖了线程1的结果

结果：counter=1，而不是预期的 2
```

这就是**竞争条件（Race Condition）**。

### 步骤

**1. 在 `main.cpp` 中写竞争条件演示（不需要新文件）**

```
演示逻辑：
1. 共享变量 int counter = 0
2. 创建 2 个线程，每个线程执行 100000 次 counter++
3. 等待两个线程都完成
4. 打印 counter 的值
```

C++ 创建线程的方式：

```cpp
#include <thread>

void increment(int& counter, int times) {
    for (int i = 0; i < times; i++) {
        counter++;    // 不加锁，故意产生竞争
    }
}

// 在 main() 里：
int counter = 0;
std::thread t1(increment, std::ref(counter), 100000);
std::thread t2(increment, std::ref(counter), 100000);
t1.join();   // 等待 t1 完成
t2.join();   // 等待 t2 完成

std::cout << "预期：200000，实际：" << counter << "\n";
```

> **`std::ref`：** 如果直接写 `std::thread t1(increment, counter, 100000)`，C++ 会拷贝 counter，线程修改的是拷贝，不影响原变量。用 `std::ref(counter)` 才是传引用。

**2. 运行多次，观察结果**

每次运行，`counter` 的值都会不同（通常小于 200000），且不固定。这就是竞争条件的特征：**结果不可预测**。

**3. 加锁修复**

加上 `std::mutex`，再次运行，确认结果稳定为 200000：

```cpp
#include <mutex>

std::mutex mtx;

void incrementSafe(int& counter, int times, std::mutex& m) {
    for (int i = 0; i < times; i++) {
        std::lock_guard<std::mutex> lock(m);   // 加锁
        counter++;
        // lock_guard 析构时自动解锁
    }
}
```

### 检查点
- [ ] 不加锁时，多次运行结果不同且小于 200000
- [ ] 加锁后，结果稳定为 200000
- [ ] 能解释：为什么 `counter++` 不是原子操作？

## 实验二：实现信号量，替代 Mutex

### 目标
理解信号量是比互斥锁更通用的同步工具，自己实现一个 `Semaphore` 类，并用它替代 `std::mutex` 完成互斥。

### 背景知识：信号量

信号量是一个整数加上两个原子操作：

| 操作 | 书上符号 | C++ 函数名 | 含义 |
|------|---------|-----------|------|
| P 操作 | `wait(S)` | `acquire()` | 值 -1；若值 < 0，阻塞等待 |
| V 操作 | `signal(S)` | `release()` | 值 +1；若有等待者，唤醒一个 |

信号量的初始值决定用途：

| 初始值 | 用途 |
|--------|------|
| 1 | 互斥锁（同时只有一个线程进入临界区） |
| 0 | 同步（一个线程等另一个线程的信号） |
| n | 管理 n 个相同资源（计数信号量） |

### 步骤

**1. 新建 `src/sync/semaphore.h`**

```
class Semaphore {
public:
    // count：信号量初始值
    explicit Semaphore(int count);

    void acquire();   // P 操作：等待（计数-1，若<0则阻塞）
    void release();   // V 操作：释放（计数+1，唤醒一个等待者）

private:
    int count;
    std::mutex mtx;
    std::condition_variable cv;
};
```

> `std::condition_variable` 是 C++ 的条件变量，用于线程等待和唤醒：
> - `cv.wait(lock, condition)`：释放锁并阻塞，直到条件为 true 且被 notify 唤醒
> - `cv.notify_one()`：唤醒一个等待的线程

**2. 新建 `src/sync/semaphore.cpp`，实现两个函数**

`acquire()` 的逻辑：
```
加锁
count--
如果 count < 0：
    cv.wait(lock)   ← 释放锁并阻塞，等待被 release() 唤醒
解锁
```

`release()` 的逻辑：
```
加锁
count++
cv.notify_one()   ← 唤醒一个等待的线程
解锁
```

**3. 用 Semaphore 重写实验一的加锁版本**

把 `std::mutex` + `std::lock_guard` 替换为 `Semaphore`（初始值为 1）：

```cpp
Semaphore sem(1);   // 二值信号量，等价于互斥锁

void incrementWithSem(int& counter, int times, Semaphore& s) {
    for (int i = 0; i < times; i++) {
        s.acquire();   // P 操作
        counter++;
        s.release();   // V 操作
    }
}
```

验证结果仍为 200000。

### 检查点
- [ ] `Semaphore` 类编译通过
- [ ] 用 Semaphore 替代 mutex 后，counter 结果仍然正确
- [ ] 能解释：`condition_variable::wait` 为什么需要传入一个锁？

## 实验三：生产者-消费者问题

### 目标
用 3 个信号量解决有界缓冲区的生产者-消费者问题：生产者满时等待，消费者空时等待，缓冲区互斥访问。

### 背景知识：三个信号量的作用

```
信号量设计（缓冲区大小 = 5）：

mutex   = Semaphore(1)    保护缓冲区的互斥访问
empty   = Semaphore(5)    空槽数量，初始=5（生产者等空位）
full    = Semaphore(0)    满槽数量，初始=0（消费者等商品）

生产者：
  empty.acquire()   ← 等一个空位（空位=0时阻塞）
  mutex.acquire()   ← 进入临界区
  放入商品
  mutex.release()   ← 离开临界区
  full.release()    ← 通知消费者有新商品

消费者：
  full.acquire()    ← 等一个商品（商品=0时阻塞）
  mutex.acquire()   ← 进入临界区
  取出商品
  mutex.release()   ← 离开临界区
  empty.release()   ← 通知生产者有新空位
```

> ⚠️ **顺序不能错：** 必须先 acquire 信号量，再 acquire mutex。反过来会死锁。

### 步骤

**1. 新建 `src/sync/producer_consumer.h`**

```
// 缓冲区大小
const int BUFFER_SIZE = 5;

// 生产者线程函数：生产 count 个商品
void producerFunc(int id, int count,
                  std::queue<int>& buffer,
                  Semaphore& mutex, Semaphore& empty, Semaphore& full);

// 消费者线程函数：消费 count 个商品
void consumerFunc(int id, int count,
                  std::queue<int>& buffer,
                  Semaphore& mutex, Semaphore& empty, Semaphore& full);

// 运行演示：3个生产者，2个消费者
void runProducerConsumer();
```

**2. 新建 `src/sync/producer_consumer.cpp`**

实现 `producerFunc`：
```
for i in 0..count:
    int item = id * 100 + i   // 商品编号
    empty.acquire()
    mutex.acquire()
    buffer.push(item)
    打印：[生产者id] 放入商品 item，缓冲区大小 buffer.size()
    mutex.release()
    full.release()
    sleep 一会儿（模拟生产耗时）
```

实现 `consumerFunc`（结构对称）。

实现 `runProducerConsumer`：
```
创建 mutex(1), empty(BUFFER_SIZE), full(0)
创建 queue<int> buffer

创建 3 个生产者线程（各生产 10 个商品）
创建 2 个消费者线程（各消费 15 个商品，共 30 = 3×10）

join 所有线程
```

**3. 在 `main.cpp` 中调用 `runProducerConsumer()`**

把这段演示放在调度算法之后，作为"并发演示"的第一个场景。

### 预期输出（顺序可能不同，因为是真实并发）

```

[生产者1] 放入商品 100，缓冲区大小：1
[生产者2] 放入商品 200，缓冲区大小：2
[消费者1] 取出商品 100，缓冲区大小：1
[生产者1] 放入商品 101，缓冲区大小：2
...
所有线程完成！
```

## 实验四：哲学家进餐问题（安全版本）

### 目标
实现哲学家进餐问题的安全解法（奇偶序号法），证明不会发生死锁。

### 背景知识：为什么会死锁？

5个哲学家围坐，每人需要左右两根筷子才能吃饭：

```
如果每人都先拿左边：
  哲学家0 拿了筷子0，等筷子1
  哲学家1 拿了筷子1，等筷子2
  哲学家2 拿了筷子2，等筷子3
  哲学家3 拿了筷子3，等筷子4
  哲学家4 拿了筷子4，等筷子0   ← 循环等待！死锁！
```

**奇偶序号法**打破循环：
- 奇数编号的哲学家：先拿左边，再拿右边
- 偶数编号的哲学家：先拿右边，再拿左边

这样不存在"所有人都拿了一根等另一根"的循环，死锁不会发生。

### 步骤

**1. 新建 `src/sync/dining_philosophers.h`**

```
const int NUM_PHILOSOPHERS = 5;

// 每位哲学家的行为（进餐 meals 次后退出）
void philosopherFunc(int id, int meals, std::mutex chopsticks[]);

// 运行演示
void runDiningPhilosophers();
```

**2. 新建 `src/sync/dining_philosophers.cpp`**

实现 `philosopherFunc`：

```
for i in 0..meals:
    打印：哲学家id 在思考...
    sleep 一会儿

    int left  = id
    int right = (id + 1) % NUM_PHILOSOPHERS

    if id % 2 == 1:   // 奇数：先左后右
        chopsticks[left].lock()
        chopsticks[right].lock()
    else:             // 偶数：先右后左
        chopsticks[right].lock()
        chopsticks[left].lock()

    打印：哲学家id 开始进餐（第i+1次）
    sleep 一会儿（进餐）

    chopsticks[left].unlock()
    chopsticks[right].unlock()
    打印：哲学家id 放下筷子

打印：哲学家id 吃饱了，退出
```

实现 `runDiningPhilosophers`：
```
std::mutex chopsticks[NUM_PHILOSOPHERS]

创建 5 个哲学家线程，各进餐 3 次
join 所有线程
```

**3. 在 `main.cpp` 调用 `runDiningPhilosophers()`**

### 检查点
- [ ] 所有哲学家都能完成 3 次进餐，程序正常退出（不卡死）
- [ ] 打印日志中，同一时刻最多有 2 个哲学家在进餐（因为 5 根筷子，最多 2 对）
- [ ] 能向别人解释：奇偶序号法为什么不会死锁？（关键：不存在循环等待）

## 常见问题

**Q：`condition_variable::wait` 为什么需要传入锁？**  
A：`wait` 需要在阻塞前释放锁（让其他线程能进入），被唤醒后再重新获取锁。如果不传锁，它就不知道要释放谁。`std::unique_lock` 是支持手动 unlock/lock 的锁类型，`std::lock_guard` 不行，所以 condition_variable 配合 `unique_lock` 使用。

**Q：线程的输出顺序每次都不一样，怎么调试？**  
A：在打印时加一把 `std::mutex print_mutex`，所有线程打印前先 lock，打印后 unlock，这样每行输出至少不会交叉。顺序还是随机的，但至少每行是完整的。

**Q：哲学家进餐演示结束后程序卡住了？**  
A：检查是否所有线程都被 `join()`。如果某个线程死锁卡住，`join()` 会永远等待。先用一个线程测试，确认奇偶法实现正确再全部启动。
