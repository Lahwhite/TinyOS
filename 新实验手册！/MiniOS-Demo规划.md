# MiniOS Demo 规划

## 项目简介

用纯 C++ 从零手写一个操作系统模拟器，无任何 OS 库依赖。

程序启动后模拟一个简化的操作系统内核运行过程：创建进程、分配内存、调度执行、读写文件、处理死锁，所有模块共享同一套数据结构，行为互相影响。

```
输入：main.cpp 中预设的进程脚本（进程数量、资源需求、执行动作）
输出：终端打印的 OS 运行日志（进程状态、内存布局、调度甘特图、死锁报告……）
```

## 项目结构

```
MiniOS/
├── CMakeLists.txt
└── src/
    ├── main.cpp                   # 入口，模拟 OS 启动和运行
    ├── kernel/
    │   └── types.h                # 全局共享类型（PCB、MemBlock 等）
    ├── process/
    │   ├── process.h              # PCB 定义、进程状态
    │   └── scheduler.h/cpp        # 调度算法（FCFS / RR / 优先级）
    ├── memory/
    │   ├── memory_manager.h/cpp   # 物理内存分区分配
    │   └── page_table.h/cpp       # 分页与虚拟内存
    ├── sync/
    │   └── semaphore.h/cpp        # 信号量（P/V 操作）
    ├── filesystem/
    │   └── filesystem.h/cpp       # 简易 inode 文件系统
    ├── io/
    │   └── disk_scheduler.h/cpp   # 磁盘调度算法
    └── deadlock/
        └── banker.h/cpp           # 银行家算法
```

## MiniOS 运行流程

```
OS 启动（main.cpp）
    │
    ├─ 初始化内存管理器（第二阶段）
    │       └─ 建立物理内存池，空闲链表
    │
    ├─ 初始化文件系统（第五阶段）
    │       └─ 创建根目录 inode
    │
    ├─ 创建初始进程（第一阶段）
    │       └─ 分配 PCB → 向内存管理器申请内存 → 加入就绪队列
    │
    ├─ 调度循环（第三阶段）
    │       ├─ 选择进程执行（FCFS / RR / 优先级）
    │       ├─ 进程执行期间可触发：
    │       │       ├─ 申请更多内存 → 内存管理器分配（第二阶段）
    │       │       ├─ 读写文件 → 文件系统 → 磁盘调度（第五阶段）
    │       │       ├─ 申请共享资源 → 信号量同步（第四阶段）
    │       │       └─ 申请多类资源 → 银行家算法（第六阶段）
    │       └─ 进程结束 → 回收 PCB 和内存
    │
    └─ 打印运行报告（各阶段逐步丰富）
```

## 分阶段规划

### 第一阶段：搭框架，OS 能跑起来

**目标：** 一个 main.cpp 能运行，创建几个进程（PCB），打印出它们的基本信息。

**涉及知识：**
- PCB（进程控制块）是什么，包含哪些字段
- 进程状态机（新建 / 就绪 / 运行 / 终止）
- CMake 多文件项目怎么组织

**结束时能回答：** 进程和程序有什么区别，PCB 里放了什么

---

### 第二阶段：内存管理

**目标：** 进程创建时向内存管理器申请内存，终止时归还，能看到内存布局变化

**涉及知识：**
- 动态分区分配（首次适应 / 最佳适应）
- 内存碎片是什么，如何合并空闲区
- 逻辑地址与物理地址的区别

**结束时能回答：** 首次适应和最佳适应的区别，外部碎片怎么产生

---

### 第三阶段：进程调度

**目标：** 多个进程竞争 CPU，用不同调度算法决定谁先运行，打印 Gantt 图

**涉及知识：**
- FCFS、SJF、RR、优先级调度
- 周转时间、等待时间的计算
- 抢占式与非抢占式的区别

**结束时能回答：** 四种调度算法各自的适用场景，RR 时间片大小的影响

---

### 第四阶段：并发与同步

**目标：** 进程之间共享资源，用信号量避免竞争，演示生产者-消费者和哲学家进餐

**涉及知识：**
- 竞争条件是什么，临界区如何保护
- 信号量的 P/V 操作
- 用 std::thread 创建真实线程

**结束时能回答：** 信号量初始值的含义，P/V 操作的顺序为什么不能随意调换

---

### 第五阶段：文件系统与磁盘调度

**目标：** 进程执行时能读写文件，文件系统有 inode 结构，读盘请求经过磁盘调度

**涉及知识：**
- inode 是什么，文件名和 inode 的关系
- 目录树结构
- SCAN 磁盘调度算法

**结束时能回答：** 为什么 inode 不存文件名，SCAN 比 FCFS 好在哪里

---

### 第六阶段（加分）：虚拟内存与死锁

**目标：** 进程地址空间超过物理内存时触发缺页，银行家算法拒绝导致死锁的资源申请

**涉及知识：**
- 分页与页面置换（LRU）
- 死锁四个必要条件
- 银行家算法的安全状态判断

**结束时能回答：** LRU 为什么优于 FIFO，银行家算法的局限性

---

## 时间总览

| 阶段 | 内容 | 里程碑 |
|------|------|--------|
| 1 | 搭框架，创建进程 | 打印出 PCB 信息，进程状态能切换 |
| 2 | 内存管理 | 进程申请/归还内存，可视化内存布局 |
| 3 | 进程调度 | 打印 Gantt 图，对比四种调度算法指标 |
| 4 | 并发同步 | 真实线程演示生产者-消费者，无竞争条件 |
| 5 | 文件系统+磁盘调度 | 进程能 mkdir/touch/write/read，磁盘请求被调度 |
| 6 | 虚拟内存+死锁（加分） | 缺页中断触发置换，银行家算法拒绝危险请求 |

**核心目标（前5阶段）：按自己节奏推进，每阶段完成后运行看到实际效果**

## 为什么整合成一个程序而不是多个独立 Lab

分散的 Lab 各自运行，看不到模块之间的联系——就像只学了发动机、轮子、方向盘，但没见过完整的汽车。

整合成一个 MiniOS，你能看到：
- 进程创建时，调度器和内存管理器同时被触发
- 进程读文件时，文件系统查 inode，磁盘调度决定读盘顺序
- 进程申请资源时，银行家算法在后台判断是否安全

这才是操作系统的真实工作方式。
