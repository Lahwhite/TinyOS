## 实验一：定义调度器接口，实现就绪队列

### 目标
新建 `Scheduler` 类，能接收进程、维护就绪队列，为后续调度算法提供统一接口。

### 背景知识：调度器在 OS 中的位置

调度器（Scheduler）是 OS 内核的核心组件，负责回答一个问题：**就绪队列里有多个进程，现在把 CPU 给谁？**

它不关心进程在干什么，只关心"谁现在能运行"和"谁应该先运行"。

调度器需要：
- 维护一个就绪队列（`READY` 状态的进程列表）
- 每次被调用时，从队列中选出一个进程运行
- 运行结束（或时间片到）后，将进程放回队列或终止

**Gantt 图**是展示调度结果的标准方式：

```
时间轴：
P1 ████████████
P2             ██████
P3                   ████████████████
P4                                   ████████
    0    5    10   15   20   25   30   35   40
```

每段色块表示某进程占用 CPU 的时间段。

### 步骤

**1. 新建 `src/process/scheduler.h`**

```
class Scheduler {
public:
    // 将进程加入就绪队列（进程状态必须是 READY）
    void addProcess(PCB* p);

    // 运行调度：选择算法，模拟所有进程执行完毕
    // algorithm: "FCFS" / "SJF" / "RR" / "Priority"
    // time_quantum: RR 算法的时间片大小（其他算法忽略此参数）
    void run(const std::string& algorithm, int time_quantum = 2);

    // 打印 Gantt 图（调用 run 后才有数据）
    void printGantt() const;

    // 打印各进程的调度统计信息
    void printStats() const;

private:
    std::vector<PCB*> ready_queue;   // 就绪队列，存指针（不拷贝 PCB）

    // Gantt 记录：每个时间单位运行的进程名
    std::vector<std::string> gantt;  // gantt[t] = 时刻 t 运行的进程名，"idle" 表示 CPU 空闲
};
```

> **为什么存指针？** 调度器操作的是 main.cpp 里创建的 PCB，修改 `remaining_time` 等字段要直接作用到原对象，存指针可以做到。存值的话修改的是拷贝。

**2. 新建 `src/process/scheduler.cpp`**

先实现 `addProcess`（把指针 push 进 `ready_queue`）和 `printGantt`：

`printGantt` 的打印逻辑：
- 合并连续相同的时间槽，输出形如 `P1(0-5)` 的段落
- 再打印一行数字时间轴

**3. 修改 `main.cpp`，接入调度器**

```cpp
#include "process/scheduler.h"

// 在所有进程 admit() 完成后：
Scheduler scheduler;
for (auto& p : processes) {
    scheduler.addProcess(&p);   // 注意取地址，传指针
}

scheduler.run("FCFS");
scheduler.printGantt();
scheduler.printStats();
```

### 检查点
- [ ] `Scheduler` 类能编译通过
- [ ] `addProcess` 后，`ready_queue` 有对应数量的指针
- [ ] `printGantt` 格式清晰，能看出每个进程运行的时间段

## 实验二：实现 FCFS 调度

### 目标
实现先来先服务算法：按进程到达就绪队列的顺序依次分配 CPU。

### 背景知识：FCFS

规则：谁先 `admit()`（进入就绪队列），谁先运行，直到运行完（`burst_time` 用完）才换下一个。

缺点：**护航效应**——如果第一个进程很长，后面所有进程都要等，即使后来的进程只需要 1 秒。

**统计指标：**

| 指标 | 计算方式 |
|------|---------|
| 完成时间 | 进程执行结束的时刻 |
| 周转时间 | 完成时间 − 到达时间 |
| 等待时间 | 周转时间 − 执行时间（burst_time） |

### 步骤

**1. 在 `scheduler.cpp` 中实现 FCFS 分支**

在 `run()` 函数里，当 `algorithm == "FCFS"` 时执行：

```
按 arrival_time 排序 ready_queue（arrival_time 相同则按 pid 排序）

current_time = 0

for 每个进程 p in ready_queue:
    如果 current_time < p->arrival_time:
        CPU 空闲，往 gantt 里填入 "idle"，直到 current_time == p->arrival_time

    p 开始运行（可以调用 p->dispatch()，但注意不要打印太多日志影响 Gantt 图清晰度）
    
    循环 p->burst_time 次：
        gantt.push_back(p->name)
        current_time++
    
    记录：p->finish_time = current_time
    p 终止（调用 p->terminate(mem)，传入内存管理器）
```

> **注意：** `terminate()` 需要传入 `MemoryManager`，因为终止时要归还内存。所以 `Scheduler::run()` 也需要接受一个 `MemoryManager&` 参数。

**2. 实现 `printStats`**

打印表格：

```
进程    到达    执行    完成    周转    等待
P1(init)  0      5       5       5       0
P2(shell) 0      3       8       8       5
P3(vim)   1      8      16      15       7
P4(gcc)   2      6      22      20      14
P5(python)3      4      26      23      19

平均周转时间：14.20   平均等待时间：9.00
```

**3. 在 `main.cpp` 中测试 FCFS**

运行后看 Gantt 图和统计信息，手算验证：P1 完成时间 = 5，周转 = 5，等待 = 0。

### 检查点
- [ ] Gantt 图显示进程按到达顺序执行
- [ ] P1 等待时间为 0（它最先到达）
- [ ] 统计表格的周转/等待时间与手算一致
- [ ] 进程终止后内存被归还（`printLayout` 验证）

## 实验三：实现 RR（轮转调度）

### 目标
实现轮转调度算法：每个进程最多连续运行 `time_quantum` 个时间单位，时间片用完后排到队尾。

### 背景知识：RR 的核心机制

RR 与 FCFS 的最大区别：进程可能不会一次跑完，而是被"切片"分多次运行。

每次时间片结束：
- 若 `remaining_time == 0`：进程终止
- 若 `remaining_time > 0`：进程重回队尾，等待下一轮

**关键细节：** 每经过一个时间单位，要检查是否有新进程到达（`arrival_time <= current_time`）并加入就绪队列。

### 步骤

**1. 在 `scheduler.cpp` 中实现 RR 分支**

RR 比 FCFS 复杂，建议用一个独立的辅助函数：

```
void runRR(int time_quantum, MemoryManager& mem)
```

算法逻辑（使用队列，而不是排好序的列表）：

```
维护一个 deque<PCB*> queue（双端队列）
将所有 arrival_time == 0 的进程加入 queue

current_time = 0

while queue 不为空 或 还有未到达的进程:
    如果 queue 为空（但还有未到达进程）:
        CPU 空闲，gantt 填 "idle"，current_time++
        检查是否有新进程到达，加入 queue
        continue

    取出 queue 队首进程 p

    实际执行时间 = min(time_quantum, p->remaining_time)
    
    循环 实际执行时间 次:
        gantt.push_back(p->name)
        current_time++
        p->remaining_time--
        检查新到达的进程（arrival_time == current_time），加入 queue 队尾
    
    if p->remaining_time == 0:
        记录 p->finish_time = current_time
        p->terminate(mem)
    else:
        将 p 放回 queue 队尾（注意：先检查新到达的进程再放回，否则顺序不对）
```

**2. 在 `main.cpp` 中对比 FCFS 和 RR**

用同一组进程，分别跑 FCFS 和 RR（时间片 = 2），打印两种算法的平均等待时间，观察差异。

### 预期 RR 结果（时间片=2，同样5个进程）

```
Gantt（RR，时间片=2）：
P1(0-2) P2(2-4) P1(4-6) P2(6-7完成) P1(7-9完成)...
```
RR 的平均等待时间通常比 FCFS 小（对短进程更友好），但并非总如此。

### 检查点
- [ ] RR 的 Gantt 图显示进程交替执行
- [ ] `remaining_time` 随着执行正确递减
- [ ] 进程执行完毕（remaining_time == 0）时才 terminate
- [ ] 新到达的进程在正确时刻加入队列

## 实验四：实现 SJF 和优先级调度（非抢占式）

### 目标
实现另外两种调度算法，与 FCFS/RR 对比，观察各算法对不同进程集的适用性。

### 背景知识

**SJF（最短作业优先）：** 在所有已到达的就绪进程中，选择 `burst_time` 最小的执行。非抢占式版本：一旦开始就不中断，直到完成。

**优先级调度：** 选择 `priority` 最小的进程（1 = 最高优先级）。规则与 SJF 几乎相同，只是选择标准变了。

两者的实现结构与 FCFS 类似，区别只在**选进程的那一步**。

### 步骤

**1. 实现 SJF**

在 `run()` 的 `"SJF"` 分支中：

```
while 还有未完成的进程:
    在所有"已到达（arrival_time <= current_time）且未完成"的进程中：
    选择 burst_time 最小的那个
    
    如果没有已到达的进程：
        gantt 填 "idle"，current_time++
        continue
    
    执行该进程（一次性跑完 burst_time，不分片）
    记录完成时间，terminate
```

**2. 实现优先级调度**

结构与 SJF 完全相同，把选择条件从 `burst_time 最小` 改为 `priority 最小`。

**3. 在 `main.cpp` 中四种算法全部运行**

```cpp
// 四种算法分别运行，每次用原始 PCB 数据的拷贝（避免状态污染）
// 方法：在 main.cpp 里定义一个辅助函数 runWithAlgorithm(algorithm, quantum)
// 每次调用前重置所有进程状态（state=NEW, remaining_time=burst_time, memory_base=-1）
```

打印四种算法的平均等待时间对比：

```
=== 调度算法对比 ===
FCFS:      平均等待时间 9.00，平均周转时间 14.20
SJF:       平均等待时间 5.40，平均周转时间 10.60  ← 通常最小
RR(q=2):   平均等待时间 7.20，平均周转时间 12.40
Priority:  平均等待时间 6.80，平均周转时间 12.00
```

### 检查点
- [ ] SJF 每次选择 burst_time 最小的就绪进程
- [ ] 优先级调度每次选择 priority 最小的就绪进程
- [ ] 四种算法的平均等待时间与手算一致
- [ ] SJF 的平均等待时间在这组数据下最小（理论上 SJF 平均等待时间最优）

## 常见问题

**Q：重置进程状态时要注意什么？**  
A：重置时需要把以下字段恢复到初始值：`state = NEW`、`remaining_time = burst_time`、`memory_base = -1`、`finish_time = 0`、`waiting_time = 0`。建议在 PCB 里写一个 `reset()` 函数统一处理。

**Q：Scheduler::run() 需要 MemoryManager 引用，但我的调度器是单独的对象，怎么传进去？**  
A：在 `run()` 的参数里加一个 `MemoryManager& mem`，从 `main.cpp` 调用时把 mem 传进去即可：`scheduler.run("FCFS", 2, mem)`。

**Q：RR 中新进程加入队列的时机很微妙，什么时候加最准确？**  
A：在每个时间单位结束后（`current_time++` 之后），立即检查是否有 `arrival_time == current_time` 的进程。注意：是在当前进程时间片内的每一步都检查，不是等整个时间片结束再检查。
