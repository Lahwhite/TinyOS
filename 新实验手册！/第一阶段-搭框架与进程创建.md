# 实验手册 · 第一阶段：搭框架，OS 能跑起来

> **阶段目标：** 编译通过，运行 `./minios`，终端打印出若干进程的基本信息，进程状态能从"新建"切换到"就绪"再到"终止"。  
> **预计时间：** 1～2 周，每天约 1 小时  
> **本阶段涉及文件：**
> ```
> MiniOS/
> ├── CMakeLists.txt          ← 本阶段创建
> └── src/
>     ├── main.cpp            ← 本阶段创建
>     └── process/
>         ├── process.h       ← 本阶段创建（PCB 定义）
>         └── scheduler.h     ← 本阶段创建（仅声明，暂不实现）
> ```
> **本阶段不涉及：** `memory/` / `sync/` / `filesystem/` / `io/` / `deadlock/`，这些留给后续阶段，现在不要创建。

## 实验一：搭建项目，让程序跑起来

### 目标
写一个最简单的 C++ 程序，用 CMake 编译，能运行并打印 `MiniOS 启动...`。

### 背景知识

CMake 是 C++ 项目最常用的构建工具。你不需要手动写编译命令，只需要写一个 `CMakeLists.txt` 描述"这个项目有哪些文件、用什么标准编译"，CMake 替你生成编译指令。

本项目最终只有**一个可执行文件** `minios`，所有阶段的代码都会逐步加进来。每完成一个阶段，就在 `CMakeLists.txt` 里追加新的 `.cpp` 文件，重新编译，程序输出的信息就会更丰富。

### 步骤

**1. 创建目录结构**

手动创建以下目录和空文件：
```
MiniOS/
├── CMakeLists.txt
└── src/
    ├── process/
    └── main.cpp
```

**2. 编写 `CMakeLists.txt`**

| 需要声明的内容 | CMake 指令 | 说明 |
|---|---|---|
| CMake 最低版本 | `cmake_minimum_required(VERSION 3.10)` | 固定写法 |
| 项目名称 | `project(MiniOS)` | 随意起名 |
| C++ 标准 | `set(CMAKE_CXX_STANDARD 17)` | 使用 C++17 |
| 查找线程库 | `find_package(Threads REQUIRED)` | 第四阶段真实线程需要，现在先写上 |
| 头文件搜索路径 | `include_directories(src)` | 让所有文件都能 `#include "process/process.h"` |
| 可执行文件 | `add_executable(minios src/main.cpp)` | 现在只有 main.cpp |
| 链接线程库 | `target_link_libraries(minios Threads::Threads)` | 配合上面的线程库 |

> ⚠️ **注意：** 后续每完成一个阶段，都需要把新的 `.cpp` 文件追加到 `add_executable` 的列表里。这一点很重要，后续阶段会反复用到。

**3. 编写 `src/main.cpp`**

先写最简单的入口：
```cpp
#include <iostream>

int main() {
    std::cout << "MiniOS 启动..." << std::endl;
    return 0;
}
```

**4. 编译并运行**

```bash
cd MiniOS
mkdir build
cd build
cmake ..
make
./minios
```

看到 `MiniOS 启动...` 输出，实验一完成。

### 检查点
- [ ] `cmake ..` 没有报错
- [ ] `make` 编译成功
- [ ] `./minios` 输出了 `MiniOS 启动...`

## 实验二：定义 PCB，让进程有"身份证"

### 目标
定义进程控制块（PCB）结构体，在 `main.cpp` 里手动创建几个进程，打印它们的信息。

### 背景知识：什么是 PCB？

操作系统同时管理很多进程，需要为每个进程记录一套信息，这就是**进程控制块（PCB，Process Control Block）**。PCB 是进程的唯一标识，进程存在则 PCB 存在，进程终止则 PCB 被销毁。

PCB 里需要记录什么？

| 字段 | 类型 | 含义 |
|------|------|------|
| `pid` | `int` | 进程唯一编号（每创建一个进程自动递增） |
| `name` | `std::string` | 进程名，便于日志识别 |
| `state` | `enum` | 当前状态（见下方状态机） |
| `priority` | `int` | 优先级，数字越小越高（第三阶段调度用） |
| `arrival_time` | `int` | 到达就绪队列的时刻（模拟时间单位） |
| `burst_time` | `int` | 总共需要 CPU 多少时间单位（执行时间） |
| `remaining_time` | `int` | 剩余还需要的 CPU 时间（RR 调度用，初始等于 burst_time） |
| `memory_size` | `int` | 需要的内存大小（KB），第二阶段内存管理用 |
| `memory_base` | `int` | 分配到的内存起始地址（第二阶段填入，初始为 -1） |

**进程状态机：**

```
NEW（新建）──→ READY（就绪）──→ RUNNING（运行）──→ TERMINATED（终止）
                   ↑                  │
                   │    等待事件完成    │
                   └──── BLOCKED ◀────┘
```

用 `enum class` 定义这 5 个状态，而不是用整数或字符串，可以让编译器帮你检查错误。

### 步骤

**1. 新建 `src/process/process.h`**

需要定义：
```
// 进程状态枚举
enum class ProcessState {
    NEW, READY, RUNNING, BLOCKED, TERMINATED
};

// 将状态转为可打印字符串的辅助函数
std::string stateToString(ProcessState s);

// PCB 结构体
struct PCB {
    // 上表中所有字段
    // 构造函数：接收 pid, name, priority, arrival_time, burst_time, memory_size
    // 在构造函数中：
    //   state = ProcessState::NEW
    //   remaining_time = burst_time
    //   memory_base = -1（-1 表示还未分配内存）
};

// 打印单个 PCB 信息的辅助函数
void printPCB(const PCB& p);
```

> **注意：** `process.h` 是纯头文件，不需要对应的 `.cpp`。把 `stateToString` 和 `printPCB` 都用 `inline` 函数在头文件里实现，或者新建一个 `process.cpp` 也可以。选一种方式，全程保持一致。

**2. 修改 `src/main.cpp`，创建几个进程**

```cpp
#include <iostream>
#include <vector>
#include "process/process.h"

int main() {
    std::cout << "MiniOS 启动...\n\n";

    // 手动创建几个进程（第三阶段前不需要自动化）
    // 参数：pid, name, priority, arrival_time, burst_time, memory_size(KB)
    std::vector<PCB> processes = {
        PCB(1, "init",   1, 0, 5, 100),
        PCB(2, "shell",  2, 0, 3, 200),
        PCB(3, "vim",    3, 1, 8, 150),
        PCB(4, "gcc",    2, 2, 6, 300),
        PCB(5, "python", 3, 3, 4,  80),
    };

    std::cout << "=== 进程列表 ===\n";
    for (auto& p : processes) {
        printPCB(p);
    }

    return 0;
}
```

**3. 编译并运行**

只需重新 `make`（不需要重新 `cmake ..`）：
```bash
cd build
make
./minios
```

### 预期输出

```
MiniOS 启动...

=== 进程列表 ===
[PID=1] init      状态:NEW   优先级:1  到达:0  执行:5  内存:100KB  内存基址:-1
[PID=2] shell     状态:NEW   优先级:2  到达:0  执行:3  内存:200KB  内存基址:-1
[PID=3] vim       状态:NEW   优先级:3  到达:1  执行:8  内存:150KB  内存基址:-1
[PID=4] gcc       状态:NEW   优先级:2  到达:2  执行:6  内存:300KB  内存基址:-1
[PID=5] python    状态:NEW   优先级:3  到达:3  执行:4  内存:80KB   内存基址:-1
```

格式不必完全一样，但每个字段都要打印出来。

- [ ] `stateToString` 能将 5 种状态转为字符串

## 实验三：实现进程状态切换

### 目标
给 PCB 加上状态切换函数，模拟进程从新建到终止的完整生命周期，并打印每次状态变化。

### 背景知识：状态切换的规则

不是任意两种状态之间都能切换，操作系统对状态切换有严格规定：

| 切换 | 触发条件 |
|------|---------|
| NEW → READY | 进程初始化完毕，进入就绪队列 |
| READY → RUNNING | 调度器选中该进程，分配 CPU |
| RUNNING → READY | 时间片用完（RR）或被高优先级进程抢占 |
| RUNNING → BLOCKED | 进程等待 I/O 或信号量 |
| BLOCKED → READY | 等待的事件完成 |
| RUNNING → TERMINATED | 进程执行完毕 |

非法切换（如 TERMINATED → RUNNING）不应该发生，代码里要检查。

### 步骤

**1. 在 `process.h` 中给 PCB 新增状态切换函数**

需要实现以下函数，每个函数要：
- 检查当前状态是否允许此切换（不合法就打印错误并返回）
- 修改 `state`
- 打印日志，格式：`[PID=1] init: NEW → READY`

```
void admit();          // NEW → READY（进程就绪）
void dispatch();       // READY → RUNNING（被调度器选中）
void preempt();        // RUNNING → READY（被抢占，时间片到）
void block();          // RUNNING → BLOCKED（等待事件）
void wakeup();         // BLOCKED → READY（事件完成，唤醒）
void terminate();      // RUNNING → TERMINATED（执行完毕）
```

**2. 修改 `main.cpp`，模拟一个简单的运行过程**

不需要真正的调度算法，手动触发状态切换即可：

```cpp
// 模拟 P1 的完整生命周期
processes[0].admit();      // NEW → READY
processes[0].dispatch();   // READY → RUNNING
processes[0].terminate();  // RUNNING → TERMINATED

// 模拟 P2 被阻塞再唤醒
processes[1].admit();
processes[1].dispatch();
processes[1].block();      // 等待 I/O
processes[1].wakeup();     // I/O 完成
processes[1].dispatch();   // 再次被调度
processes[1].terminate();
```

**3. 在 `main.cpp` 末尾打印最终状态**

循环打印所有进程的最终状态，确认状态变化被正确记录。

### 预期输出

```
MiniOS 启动...

=== 进程创建 ===
[PID=1] init    NEW
[PID=2] shell   NEW
...

=== 模拟运行 ===
[PID=1] init:  NEW → READY
[PID=1] init:  READY → RUNNING
[PID=1] init:  RUNNING → TERMINATED

[PID=2] shell: NEW → READY
[PID=2] shell: READY → RUNNING
[PID=2] shell: RUNNING → BLOCKED   （等待 I/O）
[PID=2] shell: BLOCKED → READY     （I/O 完成）
[PID=2] shell: READY → RUNNING
[PID=2] shell: RUNNING → TERMINATED

=== 最终状态 ===
[PID=1] init    TERMINATED
[PID=2] shell   TERMINATED
[PID=3] vim     NEW
...
```

### 检查点
- [ ] 6 个状态切换函数全部实现
- [ ] 非法切换（如 TERMINATED → RUNNING）会打印错误提示，不会崩溃
- [ ] `main.cpp` 能演示 P1 完整生命周期 + P2 阻塞-唤醒流程
- [ ] 最终状态打印正确

## 常见问题

**Q：`enum class` 和 `enum` 有什么区别？**  
A：`enum class` 的枚举值不会污染外层命名空间，必须写 `ProcessState::READY` 而不能直接写 `READY`。好处是不同枚举类型的值不会冲突，代码更清晰。本项目用 `enum class`。

**Q：PCB 该放在 `.h` 还是 `.cpp`？**  
A：结构体定义必须放在 `.h` 里，否则其他文件 `#include` 后看不到定义。函数的实现可以放在 `.cpp`，也可以用 `inline` 放在 `.h` 里（头文件实现）。本阶段文件少，放在 `.h` 里更简单。

**Q：`std::vector<PCB>` 会拷贝对象吗？**  
A：是的，`vector` 存储的是值的拷贝。如果用范围 for 循环修改状态，要用引用：`for (auto& p : processes)`，不加 `&` 的话修改的是拷贝，原对象不变。
