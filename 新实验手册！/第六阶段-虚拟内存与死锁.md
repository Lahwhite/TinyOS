## 实验一：理解分页，构建页表

### 目标
定义页表结构，能将进程的虚拟地址翻译为物理地址。

### 背景知识：为什么需要虚拟内存？

第二阶段的内存管理方案（动态分区分配）有两个根本缺陷：
1. 进程需要一块**连续**的物理内存，而随着分配/释放，连续的大块越来越稀缺
2. 进程大小受**物理内存**大小限制，超过就没法运行

**分页**解决了这两个问题：
- 把物理内存切成等大的**帧（Frame）**，把进程地址空间切成等大的**页（Page）**
- 页和帧大小相同，一个页可以映射到任意一个帧，不需要连续
- 进程看到的是**虚拟地址**，实际数据在物理帧里

**地址翻译：**

```
虚拟地址  =  页号(p)  +  页内偏移(d)
物理地址  =  帧号(f)  +  页内偏移(d)

其中：f = 页表[p]   （页表把页号映射到帧号）

如果页表[p] == "不在内存"：触发缺页中断（Page Fault）
```

**例子（页大小 = 4KB，进程有 4 个页）：**

| 页号 | 帧号 | 有效位 |
|------|------|--------|
| 0 | 2 | ✓ |
| 1 | 5 | ✓ |
| 2 | — | ✗ （缺页） |
| 3 | 7 | ✓ |

访问虚拟地址 `0x2001`（页号=2，偏移=1）：有效位为 ✗，触发缺页中断。

### 步骤

**1. 新建 `src/memory/page_table.h`**

```
struct PageTableEntry {
    int frame_num;   // -1 表示该页不在内存中
    bool valid;      // true = 在内存中
};

class PageTable {
public:
    // num_pages：进程的虚拟页数
    // page_size：每页大小（KB）
    PageTable(int num_pages, int page_size);

    // 翻译虚拟地址：
    //   返回物理地址（0 以上），或
    //   -1 表示缺页（page fault）
    int translate(int virtual_address);

    // 手动设置某页的映射（加载到物理帧时调用）
    void setMapping(int page_num, int frame_num);

    // 打印页表
    void printTable() const;

    int getPageSize() const;
    int getNumPages() const;

private:
    int page_size;
    std::vector<PageTableEntry> entries;
};
```

**2. 新建 `src/memory/page_table.cpp`**

实现 `translate`：
```
page_num  = virtual_address / page_size
offset    = virtual_address % page_size

if entries[page_num].valid == false:
    return -1   （缺页中断信号）

frame_num    = entries[page_num].frame_num
physical_addr = frame_num * page_size + offset
return physical_addr
```

实现 `setMapping`：直接设置 `entries[page_num]` 的 frame_num 和 valid = true。

**3. 在 `main.cpp` 中测试页表翻译**

```cpp
// 创建一个有 8 个页，每页 1KB 的页表
PageTable pt(8, 1024);

// 手动设置映射（模拟前 4 页已加载到帧 2, 5, -（缺页）, 7）
pt.setMapping(0, 2);
pt.setMapping(1, 5);
// 故意不设置 page 2（缺页）
pt.setMapping(3, 7);

pt.printTable();

// 翻译测试
int pa = pt.translate(0 * 1024 + 100);   // 页0，偏移100 → 帧2×1024+100 = 2148
std::cout << "虚拟地址 100 → 物理地址 " << pa << "\n";

pa = pt.translate(2 * 1024 + 0);         // 页2 → 缺页！
std::cout << "虚拟地址 2048 → " << (pa == -1 ? "缺页中断" : std::to_string(pa)) << "\n";
```

- [ ] 有效位为 false 时返回 -1（缺页信号）

## 实验二：缺页中断与 LRU 置换

### 目标
当进程访问一个不在内存中的页时，触发缺页中断，用 LRU 算法选择一个已在内存中的页换出，把新页换入。

### 背景知识：LRU（最近最少使用）

物理帧是有限的（例如只有 4 帧），但进程可能有很多页。当物理帧全满且发生缺页时，需要选一个现有帧换出（Evict）。

**LRU 策略：** 换出**最久没有被访问**的那一页。理论上效果好，因为符合"最近用过的将来也会用"的局部性原理。

**LRU 的实现（近似栈模拟）：**

维护一个页面访问队列，最近访问的在队列末尾，最久没用的在队列头部。

```
物理帧：[0, 1, 2, 3]（共 4 帧）
访问序列：0 1 2 3 0 1 4 0 1 2 3 4

帧满后，访问页 4 时：
  LRU 队列（头=最久）：[2, 3, 0, 1]（0和1最近用过）
  换出页 2，换入页 4
  LRU 队列变为：[3, 0, 1, 4]
```

**缺页率 = 缺页次数 / 总访问次数**，LRU 的缺页率通常低于 FIFO。

### 步骤

**1. 在 `page_table.h` 中新增 VirtualMemoryManager 类**

```
class VirtualMemoryManager {
public:
    // num_frames：物理帧数（模拟的物理内存帧）
    // num_pages：进程的虚拟页数
    // page_size：页大小（KB）
    VirtualMemoryManager(int num_frames, int num_pages, int page_size);

    // 访问虚拟地址，返回物理地址
    // 如果缺页，自动触发置换（LRU），返回置换后的物理地址
    int access(int virtual_address);

    // 打印缺页统计信息
    void printStats() const;

private:
    int num_frames;
    PageTable page_table;

    // LRU 队列：最近访问的页在队尾，最久未用的在队首
    std::list<int> lru_queue;       // 存页号
    std::unordered_map<int, int> frame_map;  // 页号 → 帧号

    // 已使用的帧（未满时不需要换出）
    int frames_used;

    int page_faults;     // 缺页次数
    int total_accesses;  // 总访问次数

    // 触发缺页处理（LRU 置换）
    int handlePageFault(int page_num);
};
```

**2. 实现 `access`**

```
page_num = virtual_address / page_size
offset   = virtual_address % page_size
total_accesses++

if page_table.translate(virtual_address) != -1:
    // 命中：更新 LRU 队列（把这一页移到队尾）
    lru_queue.remove(page_num)
    lru_queue.push_back(page_num)
    return page_table.translate(virtual_address)
else:
    // 缺页
    page_faults++
    int frame = handlePageFault(page_num)
    return frame * page_size + offset
```

**3. 实现 `handlePageFault`**

```
if frames_used < num_frames:
    // 帧未满，直接分配下一帧
    frame = frames_used++
else:
    // 帧已满，用 LRU 换出队首的页（最久未用）
    int evict_page = lru_queue.front()
    lru_queue.pop_front()
    frame = frame_map[evict_page]
    page_table 中 evict_page 的 valid 设为 false
    打印：[VM] 换出 Page evict_page（帧 frame）

// 加载新页
page_table.setMapping(page_num, frame)
frame_map[page_num] = frame
lru_queue.push_back(page_num)

打印：[VM] Page page_num 装入帧 frame（缺页第 page_faults 次）
return frame
```

**4. 在 `main.cpp` 中演示**

```cpp
// 4 帧，8 页，每页 1KB
VirtualMemoryManager vmm(4, 8, 1024);

// 访问序列（教材经典例子）
std::vector<int> accesses = {0, 1, 2, 3, 0, 1, 4, 0, 1, 2, 3, 4};
for (int page : accesses) {
    int vaddr = page * 1024;   // 访问每页的起始地址
    int paddr = vmm.access(vaddr);
    std::cout << "访问虚拟页 " << page << " → 物理地址 " << paddr << "\n";
}

vmm.printStats();
```

### 预期输出

```
[VM] Page 0 装入帧 0（缺页第1次）
[VM] Page 1 装入帧 1（缺页第2次）
[VM] Page 2 装入帧 2（缺页第3次）
[VM] Page 3 装入帧 3（缺页第4次）
访问虚拟页 0 → 物理地址 0（命中）
...
[VM] 换出 Page 2（帧 2）
[VM] Page 4 装入帧 2（缺页第5次）

## 实验三：死锁四个必要条件，复现死锁

### 目标
复现一个真实的死锁场景，终端输出卡住，然后分析四个必要条件是否全部满足。

### 背景知识：死锁四个必要条件

死锁发生，以下四个条件**缺一不可**：

| 条件 | 含义 |
|------|------|
| 互斥（Mutual Exclusion）| 资源同时只能被一个进程使用 |
| 占有并等待（Hold and Wait）| 进程持有资源并等待更多资源 |
| 不可抢占（No Preemption）| 资源不能被强制取走，只能主动释放 |
| 循环等待（Circular Wait）| 进程形成环形等待链 |

只要破坏任意一个，就能避免或消除死锁。

### 步骤

**1. 在 `main.cpp` 中复现死锁（不用新文件）**

```cpp
std::mutex resourceA, resourceB;

// 线程1：先锁 A，再锁 B
auto thread1 = [&]() {
    resourceA.lock();
    std::cout << "[T1] 获得资源A，等待资源B...\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    resourceB.lock();   // ← 死锁点：T2 拿着 B 等 A
    std::cout << "[T1] 获得资源B，开始工作\n";
    resourceB.unlock();
    resourceA.unlock();
};

// 线程2：先锁 B，再锁 A
auto thread2 = [&]() {
    resourceB.lock();
    std::cout << "[T2] 获得资源B，等待资源A...\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    resourceA.lock();   // ← 死锁点：T1 拿着 A 等 B
    std::cout << "[T2] 获得资源A，开始工作\n";
    resourceA.unlock();
    resourceB.unlock();
};
```

> ⚠️ **警告：** 这段代码会真正死锁卡住。不要放在 `main()` 的主流程中直接运行。建议注释掉，只在理解死锁时手动取消注释测试，然后用 `Ctrl+C` 中断。

**2. 分析四个必要条件**

对上面的死锁场景，对照四个条件一一分析：

```
互斥：resourceA 和 resourceB 都是互斥锁，同时只能一个线程持有 ✓
占有并等待：T1 持有 A 并等待 B，T2 持有 B 并等待 A ✓
不可抢占：mutex 不能被强制抢走，必须等持有者 unlock ✓
循环等待：T1 等 T2 持有的 B，T2 等 T1 持有的 A，构成环 ✓

四个条件全部满足 → 死锁！
```

**3. 破坏循环等待**

最简单的修复：让两个线程**按相同顺序**加锁：

```cpp
// 修复：两个线程都先锁 A，再锁 B
auto thread1Fixed = [&]() { resourceA.lock(); resourceB.lock(); /* ... */ };
auto thread2Fixed = [&]() { resourceA.lock(); resourceB.lock(); /* ... */ };
```

不再有循环等待，死锁消失（T2 会在 T1 释放 A 之后才能获得 A）。

### 检查点
- [ ] 能解释死锁复现场景中四个条件如何满足
- [ ] 修复版程序不会卡住，两个线程都能完成
- [ ] 在手册上写下：如果要破坏"互斥"条件，该怎么做？（提示：读写锁、无锁数据结构）

## 实验四：银行家算法

### 目标
实现银行家算法，在进程申请资源前判断系统是否处于安全状态，拒绝会导致死锁的申请。

### 背景知识：银行家算法

**核心思路：** 类比银行放贷——银行不会把钱全贷出去，必须留一部分准备金，保证随时能应对客户还款后的下一轮贷款。OS 资源分配同理。

**需要维护的数据（设有 n 个进程，m 类资源）：**

| 矩阵/向量 | 含义 |
|-----------|------|
| `Available[m]` | 每类资源当前剩余可用量 |
| `Max[n][m]` | 每个进程对每类资源的最大需求量 |
| `Allocation[n][m]` | 每个进程已分配的各类资源量 |
| `Need[n][m]` | 每个进程还需要的各类资源量：`Need = Max - Allocation` |

**安全状态检测（Safety Algorithm）：**

```
Work = Available
Finish[i] = false for all i

while 存在未完成的进程 i 满足 Need[i] <= Work:
    Work = Work + Allocation[i]   （模拟进程 i 执行完毕，释放资源）
    Finish[i] = true

if Finish[i] == true for all i:
    系统处于安全状态
else:
    不安全
```

**资源请求算法：**

```
进程 Pi 申请资源 Request[i]：

1. Request[i] <= Need[i]？否则报错（超过最大需求）
2. Request[i] <= Available？否则阻塞等待（资源不足）
3. 试探性分配：
     Available -= Request[i]
     Allocation[i] += Request[i]
     Need[i] -= Request[i]
4. 运行安全状态检测：
     安全 → 确认分配
     不安全 → 撤销试探，拒绝请求
```

### 步骤

**1. 新建 `src/deadlock/banker.h`**

```
class BankerAlgorithm {
public:
    // n：进程数，m：资源类型数
    // available：每类资源的初始可用量
    BankerAlgorithm(int n, int m, std::vector<int> available);

    // 设置进程 i 的最大资源需求
    void setMax(int i, std::vector<int> max);

    // 处理进程 i 的资源申请请求
    // 返回：true = 批准，false = 拒绝
    bool request(int process_id, std::vector<int> req);

    // 打印当前资源分配矩阵
    void printState() const;

private:
    int n, m;
    std::vector<int> available;
    std::vector<std::vector<int>> max_need;     // Max 矩阵
    std::vector<std::vector<int>> allocation;   // Allocation 矩阵
    std::vector<std::vector<int>> need;         // Need 矩阵

    // 安全状态检测：返回是否安全，并输出安全序列
    bool isSafe() const;
};
```

**2. 新建 `src/deadlock/banker.cpp`**

实现 `isSafe`（严格按照安全状态检测伪代码）：
```
vector<int> work = available
vector<bool> finish(n, false)
vector<int> safe_sequence

while 循环至少能找到一个满足 Need[i] <= Work 的未完成进程:
    找到这样的进程 i
    Work += Allocation[i]
    finish[i] = true
    safe_sequence.push_back(i)

if 所有 finish[i] 都是 true:
    打印：安全序列 P0 → P1 → P3 → ...
    return true
else:
    打印：系统不安全，无法找到安全序列
    return false
```

实现 `request`：
```
1. 检查 req <= need[pid]（不能超过最大声明）
2. 检查 req <= available（资源够不够）
3. 试探分配：
     for j in 0..m: available[j] -= req[j]; allocation[pid][j] += req[j]; need[pid][j] -= req[j]
4. 运行 isSafe()
5. 如果安全：打印批准，return true
   如果不安全：回滚，打印拒绝，return false
```

**3. 在 `main.cpp` 中演示（教材经典例子）**

```cpp
// 3类资源，5个进程，资源总量 A=10, B=5, C=7

// 设置最大需求（教材 Table 7.2 的数据）
banker.setMax(0, {7, 5, 3});
banker.setMax(1, {3, 2, 2});
banker.setMax(2, {9, 0, 2});
banker.setMax(3, {2, 2, 2});
banker.setMax(4, {4, 3, 3});

// 设置当前已分配（教材数据）
// ... （用 request 或直接在内部设置初始 Allocation）

banker.printState();

// P1 申请资源 (1, 0, 2)
std::cout << "\nP1 申请资源 [1, 0, 2]：\n";
banker.request(1, {1, 0, 2});

// P4 申请资源 (3, 3, 0)（应被拒绝，会导致不安全）
std::cout << "\nP4 申请资源 [3, 3, 0]：\n";
banker.request(4, {3, 3, 0});
```

这是一个完整的操作系统知识地图，每一个模块都对应《操作系统（第六版）》的核心章节。

## 常见问题

**Q：LRU 实现里，`std::list::remove` 的时间复杂度是 O(n)，有没有更好的方法？**  
A：实际 LRU 缓存常见的高效实现是 `std::list` + `std::unordered_map`（记录每个页在 list 里的迭代器），这样删除和插入都是 O(1)。本阶段数据量小，O(n) 可接受，但你知道有这个优化即可。

**Q：银行家算法有什么局限性，为什么现实中 OS 不普遍使用？**  
A：三个限制：1）进程必须事先声明最大资源需求（Max），实际中不总能知道；2）进程数和资源类型数必须固定，动态创建进程的系统很难处理；3）算法本身有开销（O(n²·m)）。所以现实中更多用"鸵鸟策略"（假装死锁不存在，出了问题再重启）或用"死锁检测+恢复"。

**Q：如果我的 `isSafe` 实现总是返回 true，问题在哪里？**  
A：最常见的 bug 是 `Need[i] <= Work` 的比较写成了逐元素"全都小于等于"，但循环时条件写错了（例如写成"任意一个小于等于"）。仔细检查向量比较的逻辑，确保是所有分量都满足。
