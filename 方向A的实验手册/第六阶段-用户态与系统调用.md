# 实验手册 · TinyOS 第六阶段：用户态，保护模式真正发挥作用

> **阶段目标：** 建立 Ring 0（内核态）和 Ring 3（用户态）的隔离，实现系统调用（`int 0x80`），让用户程序能安全地请求内核服务而不能直接访问硬件。  
> **预计时间：** 2～3 周，每天约 1 小时  
> **前置条件：** 第五阶段完成，所有模块正常工作  
> **本阶段新增/修改文件：**
> ```
> TinyOS/
> ├── kernel/
> │   ├── gdt.h/c        ← 修改 GDT，加入 TSS（Task State Segment）
> │   ├── syscall.h/c    ← 系统调用分发器
> │   └── tss.h/c        ← TSS（任务状态段），Ring 切换必需
> ├── user/
> │   ├── user_entry.asm ← 用户程序汇编入口
> │   ├── user_lib.h/c   ← 用户态库（封装系统调用）
> │   └── shell.c        ← 第一个用户程序：简单 Shell
> └── linker_user.ld     ← 用户程序链接脚本
> ```

---

## 实验一：理解 Ring 0 和 Ring 3

### 背景知识：x86 特权级

x86 定义了 4 个特权级（Ring 0～3），数字越小权限越高：

```
Ring 0（内核态）：最高权限
  ✅ 执行任何指令（包括 in/out、cli/sti、lgdt、lidt）
  ✅ 访问任意物理内存
  ✅ 修改页表、段表、IDT

Ring 3（用户态）：最低权限
  ❌ 不能执行特权指令（执行 in/out 触发 GPF 异常）
  ❌ 不能直接访问 I/O 端口（无法写 VGA 显存）
  ❌ 不能访问内核内存（页表标记为非用户可访问）
  ✅ 只能通过系统调用请求内核服务
```

**前五个阶段的所有代码都运行在 Ring 0**——这意味着任何一个"进程"出 bug 都可能写乱内核数据，没有任何保护。

**本阶段目标：** 让用户程序运行在 Ring 3，让内核运行在 Ring 0，两者完全隔离。

### Ring 切换机制

**Ring 3 → Ring 0（向内：系统调用）：**
```
用户程序执行 int 0x80
CPU 自动：
  1. 从 TSS（任务状态段）里读取内核栈地址
  2. 把用户态的 SS/ESP/EFLAGS/CS/EIP 保存到内核栈
  3. 切换到内核态代码段（Ring 0）
  4. 跳转到 IDT 中 0x80 号中断的处理函数
```

**Ring 0 → Ring 3（向外：iret 返回用户态）：**
```
内核执行 iret
CPU 从栈上读取 EIP/CS/EFLAGS/ESP/SS
如果 CS 的 RPL 字段（低2位）= 3，则：
  切换到 Ring 3，恢复用户态 SS 和 ESP
  跳转到用户程序的下一条指令
```

## 实验二：修改 GDT，加入用户态段和 TSS

### 目标
在 GDT 里添加用户态代码段、用户态数据段、TSS 描述符，为 Ring 切换做准备。

### 步骤

**1. 理解 GDT 选择子的 RPL 字段**

段选择子格式（16位）：
```
位 15-3：GDT 索引（第几个描述符）
位 2：TI 位（0=GDT，1=LDT，通常用0）
位 1-0：RPL（请求特权级，0=Ring0，3=Ring3）
```

所以：
```
0x08 = 0b00001000 → GDT 第1项，RPL=0（内核代码段）
0x10 = 0b00010000 → GDT 第2项，RPL=0（内核数据段）
0x18 = 0b00011000 → GDT 第3项，RPL=0 （用户代码段，设置 DPL=3）
      加上 0x03 → 0x1B = 0b00011011（RPL=3，请求 Ring 3 权限）
0x20 = 0b00100000 → GDT 第4项，RPL=0（用户数据段，DPL=3）
      加上 0x03 → 0x23 = 0b00100011（RPL=3）
0x28 = 0b00101000 → GDT 第5项，TSS 描述符
```

**2. 修改 `kernel/gdt.c`**

在 GDT 数组里追加三个描述符：

```c
// GDT 第3项：用户代码段（DPL=3）
// 和内核代码段完全相同，只是 type_attr 的 DPL 字段改为 11（Ring 3）
// 0xFA = 11111010b：P=1, DPL=11, S=1, Type=1010（代码，可读可执行）
gdt[3] = make_gdt_entry(0x00000000, 0xFFFFF, 0xFA, 0xCF);

// GDT 第4项：用户数据段（DPL=3）
// 0xF2 = 11110010b：P=1, DPL=11, S=1, Type=0010（数据，可读可写）
gdt[4] = make_gdt_entry(0x00000000, 0xFFFFF, 0xF2, 0xCF);

// GDT 第5项：TSS 描述符（特殊格式，见下方）
// （TSS 是一个内存结构，描述符记录它的地址和大小）
```

**3. 新建 `kernel/tss.h/c`**

**TSS（任务状态段）：** CPU 从 Ring 3 切换到 Ring 0 时，需要知道内核栈在哪里。这个信息存在 TSS 里。

```c
typedef struct {
    uint32_t prev_tss;   // 前一个 TSS（多任务切换用，我们不用）
    uint32_t esp0;       // Ring 0 栈指针（切换到内核态时 CPU 用这个）
    uint32_t ss0;        // Ring 0 栈段选择子（固定 0x10）
    // ... 后续字段对我们无关，全填0
    uint32_t unused[22];
    uint16_t reserved;
    uint16_t iomap_base; // I/O 位图基地址（设为 sizeof(tss) 表示无位图）
} __attribute__((packed)) tss_t;

static tss_t tss;

void tss_init() {
    // 注册到 GDT 第5项
    uint32_t base  = (uint32_t)&tss;
    uint32_t limit = sizeof(tss) - 1;
    
    // 设置 GDT 第5项（TSS 描述符格式和普通段描述符略有不同）
    // type_attr = 0x89 = 10001001b：P=1, DPL=0, S=0, Type=1001（可用32位TSS）
    gdt_set_tss_entry(5, base, limit);
    
    tss.ss0  = 0x10;   // 内核数据段
    tss.esp0 = 0;      // 之后每次调度时更新（切换进程时要更新此字段）
    tss.iomap_base = sizeof(tss_t);
    
    // 加载 TSS：把 GDT 第5项的选择子加载到 TR 寄存器
    asm volatile("ltr %%ax" : : "a"(0x2B));  // 0x28 | 3（RPL=3，但TSS描述符本身DPL=0）
    // 注：ltr 的选择子应为 0x28（GDT第5项，偏移5*8=40=0x28），有时需要调整
}

void tss_set_kernel_stack(uint32_t stack) {
    tss.esp0 = stack;   // 每次切换进程时，用新进程的内核栈更新此字段
}
```

### 检查点
- [ ] 修改 GDT 后内核仍能正常运行（VGA 打印、中断正常）
- [ ] TSS 加载成功（`ltr` 不触发 GPF）
- [ ] 能解释：为什么 Ring 3 → Ring 0 切换时 CPU 需要从 TSS 里读内核栈地址？

## 实验三：实现系统调用（int 0x80）

### 目标
注册 `int 0x80` 的 IDT 条目（DPL=3，允许用户态触发），实现系统调用分发器，支持 `sys_write`（打印字符串）和 `sys_read`（从文件读取）。

### 背景知识：Linux 系统调用的约定

Linux 的系统调用约定（cdecl 风格，x86 32位）：

```
EAX = 系统调用号
EBX = 第1个参数
ECX = 第2个参数
EDX = 第3个参数
执行 int 0x80
EAX = 返回值
```

我们的 TinyOS 采用相同约定（这也是为什么学完这个阶段，理解 Linux 系统调用会很容易）。

**定义系统调用号：**

```c
#define SYS_EXIT    0    // 退出进程
#define SYS_WRITE   1    // 写文件/打印
#define SYS_READ    2    // 读文件
#define SYS_OPEN    3    // 打开文件
#define SYS_CLOSE   4    // 关闭文件
#define SYS_MKDIR   5    // 创建目录
#define SYS_GETPID  6    // 获取进程 ID
```

### 步骤

**1. 在 `kernel/idt.c` 里注册 0x80 号中断**

```c
// 0x80 号中断 DPL 必须是 3，才允许用户态代码触发
// type_attr = 0xEE = 11101110b：P=1, DPL=11（Ring3可触发）, Type=1110（32位中断门）
void idt_set_syscall_gate(int num, uint32_t handler) {
    idt[num].offset_low  = handler & 0xFFFF;
    idt[num].selector    = 0x08;
    idt[num].zero        = 0;
    idt[num].type_attr   = 0xEE;    // 注意：DPL=3
    idt[num].offset_high = (handler >> 16) & 0xFFFF;
}

// 在 idt_init() 里调用：
idt_set_syscall_gate(0x80, (uint32_t)syscall_entry);
```

**2. 新建 `kernel/syscall.h/c`**

```c
// syscall.h
void syscall_init();

// syscall.c
// 系统调用分发器
uint32_t syscall_dispatch(uint32_t eax, uint32_t ebx, uint32_t ecx, uint32_t edx) {
    switch (eax) {
        case SYS_EXIT:
            // 终止当前进程（调用调度器移除当前进程）
            process_exit();
            return 0;

        case SYS_WRITE:
            // ebx = 文件描述符（1 = stdout = VGA），ecx = 字符串地址，edx = 长度
            if (ebx == 1) {
                // stdout：直接打印到 VGA
                char* str = (char*)ecx;
                for (uint32_t i = 0; i < edx; i++) vga_putchar(str[i]);
                return edx;
            } else {
                // 其他：写到文件系统（调用 fs_write）
                return fs_write_fd(ebx, (uint8_t*)ecx, edx);
            }

        case SYS_READ:
            return fs_read_fd(ebx, (uint8_t*)ecx, edx);

        case SYS_GETPID:
            return scheduler_current()->pid;

        default:
            vga_puts("[kernel] Unknown syscall\n");
            return (uint32_t)-1;
    }
}
```

**3. 在 `kernel/isr.asm` 里加入系统调用入口**

```nasm
; 系统调用入口（int 0x80）
global syscall_entry
[extern syscall_dispatch]

syscall_entry:
    ; 此时 CPU 已经自动切换到 Ring 0，并从 TSS 加载了内核栈
    ; 保存用户态寄存器
    push edx
    push ecx
    push ebx
    push eax
    
    ; 调用 C 分发器（参数通过寄存器传递，这里转成 C 调用约定的栈参数）
    call syscall_dispatch
    ; 返回值在 EAX 里（C 函数默认用 EAX 返回整数）
    
    add esp, 16    ; 弹出 4 个参数
    
    iret           ; 返回用户态（恢复 Ring 3、用户 SS/ESP/EFLAGS）
```

### 检查点
- [ ] 从内核代码（Ring 0）执行 `int 0x80` 能调用 `syscall_dispatch`
- [ ] `SYS_WRITE` 能打印字符串
- [ ] `SYS_GETPID` 返回正确的 PID
- [ ] 如果用户程序调用未定义的系统调用号，内核打印错误并优雅处理（不崩溃）

## 实验四：创建用户程序，在 Ring 3 运行

### 目标
写一个简单的用户程序（不能直接调用内核函数），编译到独立的二进制文件，内核将其加载并以 Ring 3 运行。

### 步骤

**1. 新建 `user/user_lib.h/c`（用户态库，封装系统调用）**

```c
// user_lib.h
// 用户程序只能通过这些函数和内核交互，不能直接调用内核函数

static inline int sys_write_stdout(const char* str, int len) {
    int ret;
    asm volatile(
        "int $0x80"
        : "=a"(ret)
        : "a"(1),         // SYS_WRITE
          "b"(1),         // fd=1（stdout）
          "c"((uint32_t)str),
          "d"((uint32_t)len)
    );
    return ret;
}

static inline void sys_exit() {
    asm volatile(
        "mov $0, %%eax\n"   // SYS_EXIT
        "int $0x80\n"
        : : : "eax"
    );
}

// 封装成更友好的函数
void print(const char* str);   // 内部调用 sys_write_stdout
int  getpid();                  // 内部调用 SYS_GETPID
```

**为什么不能直接调用 `vga_putchar`？**

`vga_putchar` 的代码在内核内存（Ring 0 页），用户程序（Ring 3）访问这段内存会触发保护错误（General Protection Fault）。用户程序必须通过 `int 0x80` 进入内核，由内核执行 `vga_putchar`。这是"用户程序不能写 VGA 显存，只能请求内核打印"的底层原因。

**2. 新建 `user/shell.c`——第一个用户程序**

```c
// 注意：不能 #include 内核的任何头文件
// 只能包含 user_lib.h（用户态库）
#include "user_lib.h"

void user_main() {
    print("Welcome to TinyOS Shell!\n");
    print("This is running in Ring 3 (user mode).\n");
    
    int pid = getpid();
    // 打印 pid（需要用户态版的 itoa）
    print("My PID: ");
    // ... print pid
    print("\n");
    
    // 简单的"命令行"（循环读取键盘输入）
    char buf[64];
    while (1) {
        print("> ");
        // sys_read(0, buf, 64)   ← 从 stdin 读一行（需要实现 SYS_READ 和键盘队列）
        // 暂时只是打印提示符
    }
}
```

**3. 新建 `user/user_entry.asm`**

```nasm
[bits 32]
[extern user_main]

global _user_start
_user_start:
    call user_main
    
    ; user_main 不应该返回，但以防万一：调用 sys_exit
    mov eax, 0     ; SYS_EXIT
    int 0x80
    jmp $          ; 如果 exit 失败，死循环
```

**4. 新建 `linker_user.ld`**

```ld
ENTRY(_user_start)

SECTIONS {
    . = 0x400000;    /* 用户程序加载到虚拟地址 4MB（和内核隔离） */
    
    .text   : { *(.text)   }
    .rodata : { *(.rodata) }
    .data   : { *(.data)   }
    .bss    : { *(.bss)    }
}
```

**用户程序从虚拟地址 0x400000 开始**，内核在低地址（0x10000）。通过页表，可以把 0x400000 的虚拟地址映射到任意物理地址，并设置页表项 `USER` 标志位（第2位）为 1，允许 Ring 3 访问。

**5. 内核加载用户程序**

将用户程序编译成 ELF 或纯 binary，嵌入到内核磁盘镜像里（放在内核后面的扇区）。内核用 `ata_read_sector` 读取，用 `vmm_map` 映射到虚拟地址，然后切换到 Ring 3：

```c
void kernel_launch_user(uint32_t entry_virt) {
    // 更新 TSS 的内核栈（Ring 3→Ring 0 切换时使用）
    tss_set_kernel_stack(current_proc->kernel_stack_top);
    
    // 用 iret 伪造一个"从 Ring 3 中断返回"的场景，让 CPU 切换到 Ring 3
    // 手动构造中断返回栈帧：
    asm volatile(
        "mov $0x23, %%ax\n"    // 用户数据段选择子（GDT第4项 | RPL=3 = 0x20|3 = 0x23）
        "mov %%ax, %%ds\n"
        "mov %%ax, %%es\n"
        "mov %%ax, %%fs\n"
        "mov %%ax, %%gs\n"
        
        "push $0x23\n"         // SS（用户栈段）
        "push %1\n"            // ESP（用户栈指针）
        "pushf\n"              // EFLAGS（保持当前标志，包含 IF=1）
        "push $0x1B\n"         // CS（用户代码段选择子：GDT第3项 | RPL=3 = 0x18|3 = 0x1B）
        "push %0\n"            // EIP（用户程序入口）
        "iret\n"               // 切换到 Ring 3！
        : : "r"(entry_virt), "r"(user_stack_top)
    );
}
```

### 检查点
- [ ] 用户程序运行在 Ring 3（用户程序里用 `mov eax, cr0` 会触发 GPF，证明权限隔离）
- [ ] `print("Hello")` 通过 `int 0x80` 调用内核，打印到屏幕
- [ ] 用户程序企图直接写 `0xB8000`（VGA 显存）会触发页错误（因为页表没有用户权限）

## 实验五：完整 Shell，整合所有模块

### 目标
实现一个能接收键盘输入、执行简单命令的 Shell，综合运用所有前五个阶段的能力。

### Shell 支持的命令

| 命令 | 功能 |
|------|------|
| `help` | 列出所有命令 |
| `ls [path]` | 列出目录（调用 `SYS_LS`）|
| `cat [file]` | 打印文件内容（调用 `SYS_READ`）|
| `echo [text] > [file]` | 写文件（调用 `SYS_WRITE`）|
| `mkdir [dir]` | 创建目录（调用 `SYS_MKDIR`）|
| `ps` | 列出所有进程（调用 `SYS_PS`）|
| `exit` | 退出 Shell（调用 `SYS_EXIT`）|

### 步骤

**1. 实现键盘输入队列（在内核里）**

第二阶段的键盘中断只是立刻打印字符。要支持 Shell，需要一个缓冲区：键盘中断把字符放入队列，`SYS_READ fd=0` 从队列取字符。

```c
// 在 drivers/keyboard.c 里加：
#define KB_BUFFER_SIZE 64
static char kb_buffer[KB_BUFFER_SIZE];
static int  kb_head = 0, kb_tail = 0;

// 键盘中断里：
kb_buffer[kb_tail] = ascii;
kb_tail = (kb_tail + 1) % KB_BUFFER_SIZE;

// SYS_READ(fd=0) 里：
while (kb_head == kb_tail) { asm("hlt"); }  // 等待有字符
char c = kb_buffer[kb_head];
kb_head = (kb_head + 1) % KB_BUFFER_SIZE;
```

**2. 在 `user/shell.c` 里实现命令解析**

```c
void user_main() {
    print("TinyOS Shell v0.1\n");
    print("Type 'help' for available commands.\n");
    
    char line[128];
    while (1) {
        print("$ ");
        readline(line, sizeof(line));   // 从 stdin 读一行（用 sys_read 实现）
        
        if (strcmp(line, "help") == 0) {
            print("Commands: help ls cat echo mkdir ps exit\n");
        } else if (strncmp(line, "ls", 2) == 0) {
            char* path = (line[2] == ' ') ? line + 3 : "/";
            sys_ls(path);
        } else if (strncmp(line, "cat", 3) == 0) {
            char* path = line + 4;
            // ...
        } else if (strcmp(line, "ps") == 0) {
            sys_ps();
        } else if (strcmp(line, "exit") == 0) {
            print("Goodbye!\n");
            sys_exit();
        } else {
            print("Unknown command: ");
            print(line);
            print("\n");
        }
    }
}
```

**3. 追加必要的系统调用**

```c
// syscall.c 追加
case SYS_LS:     return sys_ls_impl((char*)ebx);
case SYS_PS:     return sys_ps_impl();
case SYS_READ_STDIN:   /* 从键盘队列读 */
```

## 阶段总结

**完成本阶段，TinyOS 具备了真实操作系统的核心架构：**

```
┌─────────────────────────────────────────────────────┐
│                  用户程序（Ring 3）                    │
│   Shell    │  其他程序  │  ...                        │
│   ↑ int 0x80 系统调用 ↓                              │
├─────────────────────────────────────────────────────┤
│                  内核（Ring 0）                        │
│  系统调用层  │  进程调度  │  内存管理  │  文件系统      │
│  ↑ 硬件中断 ↓                                        │
├─────────────────────────────────────────────────────┤
│                  硬件（QEMU）                          │
│   CPU   │  内存  │  磁盘  │  VGA  │  键盘  │  时钟    │
└─────────────────────────────────────────────────────┘
```

**各阶段联系总结：**

| 阶段 | 模块 | 与其他阶段的联系 |
|------|------|----------------|
| 1 引导 | `boot.asm`、`vga.c` | 提供内核运行环境，所有阶段依赖 |
| 2 中断 | `idt.c`、`pic.c`、`timer.c` | 驱动调度（第4阶段）、接收磁盘完成（第5阶段）、系统调用入口（第6阶段） |

## 常见问题

**Q：切换到 Ring 3 后立刻 GPF？**  
A：检查 CS 选择子是否正确（`0x1B`），它必须指向 GDT 里 DPL=3 的代码段。如果选择子对应的段 DPL=0，CPU 会拒绝进入 Ring 3。

**Q：用户程序里调用 `int 0x80` 触发 GPF？**  
A：确认 IDT 第 0x80 项的 `type_attr` 是 `0xEE`（DPL=3，允许 Ring 3 触发）。如果是 `0x8E`（DPL=0），Ring 3 触发会被 CPU 拒绝并产生 GPF。

**Q：用户程序访问了不该访问的内存，但没有页错误？**  
A：确认用户程序的页表项没有设置 `PAGE_USER` 标志，或者检查是否开启了分页（`vmm_init` 是否调用）。如果页表对所有地址都设置了 `PAGE_USER`，那就没有内存保护。

**Q：Shell 读取键盘后无响应，像卡住了？**  
A：键盘中断可能被阻塞（键盘 IRQ1 被屏蔽，或 `pic_send_eoi` 没有调用）。确认键盘中断处理函数在读完字符后调用了 `pic_send_eoi(1)`。
