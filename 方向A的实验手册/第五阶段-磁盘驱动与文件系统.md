# 实验手册 · TinyOS 第五阶段：磁盘驱动 + 文件系统，文件真正存到磁盘

> **阶段目标：** 实现 ATA 磁盘驱动，在 QEMU 虚拟磁盘上读写扇区；在此基础上实现 inode 文件系统（从 MiniOS 移植），文件内容真正持久化。  
> **预计时间：** 2～3 周，每天约 1 小时  
> **前置条件：** 第四阶段完成，`kmalloc` 可用，进程调度工作  
> **本阶段新增文件：**
> ```
> TinyOS/
> ├── drivers/
> │   └── ata.h/c        ← ATA/IDE 磁盘驱动
> └── filesystem/
>     ├── block.h/c      ← 块缓存（Page Cache 的简化版）
>     ├── inode.h/c      ← inode 数据结构与操作
>     └── fs.h/c         ← 文件系统 API（open/read/write/mkdir）
> ```

## 实验一：ATA 磁盘驱动，读写原始扇区

### 目标
通过 x86 I/O 端口直接操作 ATA/IDE 磁盘控制器，实现 `ata_read_sector` 和 `ata_write_sector`。

### 背景知识：真实磁盘访问 vs MiniOS 的磁盘调度

**MiniOS 第五阶段的磁盘调度（SCAN/FCFS）：**
```cpp
// 只是算法模拟：给定一组"请求"，计算磁头移动距离
disk.scan(requests);   // 输出：总移动距离 400 个磁道
// 没有真正读写任何数据
```

**TinyOS 的磁盘驱动：**
```c
// 真正通过 I/O 端口发指令给磁盘控制器
ata_read_sector(0, buffer);  // 读磁盘第0扇区的 512 字节到 buffer
ata_write_sector(0, data);   // 把 data 的 512 字节写入磁盘第0扇区
```

**ATA（IDE）接口：**

ATA 是 PC 时代最常见的磁盘接口（SATA 的前身）。QEMU 默认模拟 ATA 磁盘。ATA 通过 I/O 端口操作：

| 端口 | 名称 | 说明 |
|------|------|------|
| `0x1F0` | 数据寄存器 | 读写扇区数据（16位） |
| `0x1F1` | 错误寄存器 | 最后一次操作的错误信息 |
| `0x1F2` | 扇区数量寄存器 | 要读写的扇区数 |
| `0x1F3` | LBA 低8位 | 逻辑块地址（0x1F3-0x1F5共24位） |
| `0x1F4` | LBA 中8位 | |
| `0x1F5` | LBA 高8位 | |
| `0x1F6` | 磁盘/磁头选择寄存器 | 选主盘/从盘，LBA 第24-27位 |
| `0x1F7` | 状态/命令寄存器 | 发命令，读状态 |

**LBA（逻辑块地址）：** 把磁盘所有扇区按顺序编号（0, 1, 2, ...），比 CHS（柱面/磁头/扇区）寻址简单得多。

### 步骤

**1. 新建 `drivers/ata.h`**

```c
#ifndef ATA_H
#define ATA_H

#include <stdint.h>

#define SECTOR_SIZE 512

void    ata_init();
int     ata_read_sector (uint32_t lba, uint8_t* buffer);   // 读一个扇区
int     ata_write_sector(uint32_t lba, const uint8_t* data); // 写一个扇区

// 批量版本（更高效）
int     ata_read_sectors (uint32_t lba, uint32_t count, uint8_t* buffer);
int     ata_write_sectors(uint32_t lba, uint32_t count, const uint8_t* data);

#endif
```

**2. 新建 `drivers/ata.c`**

`ata_read_sector` 的实现步骤（固定套路）：

```c
int ata_read_sector(uint32_t lba, uint8_t* buffer) {
    // 1. 等待磁盘空闲（读 0x1F7，直到 BSY 位 = 0）
    while (inb(0x1F7) & 0x80);   // 0x80 = BSY bit

    // 2. 发送参数
    outb(0x1F6, 0xE0 | ((lba >> 24) & 0x0F));  // 选主盘，LBA 第24-27位
    outb(0x1F2, 1);               // 读1个扇区
    outb(0x1F3, lba & 0xFF);      // LBA 低8位
    outb(0x1F4, (lba >> 8) & 0xFF);  // LBA 中8位
    outb(0x1F5, (lba >> 16) & 0xFF); // LBA 高8位

    // 3. 发送 READ SECTORS 命令
    outb(0x1F7, 0x20);

    // 4. 等待数据就绪（DRQ bit = 1）
    while (!(inb(0x1F7) & 0x08));  // 0x08 = DRQ bit

    // 5. 从数据端口读取 256 个 16 位字（= 512 字节）
    uint16_t* buf16 = (uint16_t*)buffer;
    for (int i = 0; i < 256; i++) {
        buf16[i] = inw(0x1F0);    // inw = 读16位
    }
    return 0;   // 成功
}
```

`ata_write_sector` 类似，命令改为 `0x30`（WRITE SECTORS），用 `outw` 替代 `inw`。

**注意：** 需要在 `pic.h` 里追加 `inw`/`outw` 的内联实现。

**3. 在 `kernel_main` 里测试读写**

```c
uint8_t buf[512];

// 写入测试数据到扇区 100（避开前几个扇区，那里是引导扇区和内核）
uint8_t test_data[512];
memset(test_data, 0xAB, 512);
ata_write_sector(100, test_data);

// 读回来
ata_read_sector(100, buf);

// 验证
if (buf[0] == 0xAB && buf[255] == 0xAB) {
    vga_puts("ATA read/write: OK\n");
} else {
    vga_puts("ATA read/write: FAILED\n");
}
```

**4. 配置 QEMU 虚拟磁盘**

修改 Makefile，让 QEMU 同时使用软盘镜像（系统盘）和 IDE 硬盘镜像（数据盘）：

```makefile
# 创建虚拟硬盘镜像（10MB）
hdd.img:
    dd if=/dev/zero of=hdd.img bs=1024 count=10240

run: tinyos.img hdd.img
    $(QEMU) -drive format=raw,if=floppy,file=tinyos.img -hda hdd.img
```

### 检查点
- [ ] 读写扇区测试通过（写入 0xAB，读回来也是 0xAB）
- [ ] 能解释 LBA 寻址和 CHS 寻址的区别
- [ ] 能解释为什么等待 BSY 位清 0 再发命令（磁盘有机械延迟）

## 实验二：块缓存，联系磁盘和文件系统

### 目标
实现一个简单的块缓存层（类似 Linux 的 Page Cache 的简化版），文件系统通过块缓存访问磁盘，减少直接磁盘操作次数。

### 背景知识：为什么需要块缓存？

磁盘很慢（机械磁盘约 10ms 延迟），内存很快（约 100ns）。每次文件读写都直接访问磁盘是灾难性的。

**块缓存**（Block Cache / Buffer Cache）：
- 把最近读过的磁盘块缓存在内存里
- 下次访问同一块时直接返回内存里的数据
- 只有缓存被驱逐时才写回磁盘（Write-back 策略）

这就是 MiniOS 第五阶段"磁盘调度"和"文件系统"没有联系的原因——MiniOS 的文件系统直接操作内存，没有磁盘层；磁盘调度只是算法演示。TinyOS 在这里把两者联系起来。

### 步骤

**1. 新建 `filesystem/block.h`**

```c
#ifndef BLOCK_H
#define BLOCK_H

#include <stdint.h>

#define BLOCK_SIZE    512      // 等于扇区大小
#define BLOCK_CACHE_SIZE 64    // 最多缓存 64 个块

typedef struct {
    uint32_t lba;           // 这个块对应的磁盘扇区号
    uint8_t  data[BLOCK_SIZE];
    uint8_t  dirty;         // 1 = 内容被修改过，需要写回磁盘
    uint8_t  valid;         // 1 = 缓存有效
    int      last_used;     // 用于 LRU 淘汰（复用 MiniOS 的 LRU 算法！）
} block_cache_t;

void     block_init();
uint8_t* block_read(uint32_t lba);    // 返回指向缓存块数据的指针
void     block_write(uint32_t lba, const uint8_t* data);  // 写入缓存（标记 dirty）
void     block_flush();               // 把所有 dirty 块写回磁盘

#endif
```

**2. 新建 `filesystem/block.c`**

实现 `block_read`：
```
在缓存中查找 lba（遍历 cache 数组，找 valid=1 且 lba 匹配的项）
    如果找到：更新 last_used，直接返回指针（缓存命中）
    如果没找到：
        找一个空闲槽或 LRU 最久未使用的槽（复用 MiniOS LRU 算法）
        如果该槽 dirty：先调用 ata_write_sector 写回磁盘
        调用 ata_read_sector 从磁盘读取数据
        更新缓存：valid=1, dirty=0, lba=目标lba
        返回数据指针
```

**LRU 驱逐：** 维护一个全局 `timer` 计数器，每次访问缓存时更新被访问块的 `last_used = timer++`。驱逐时选 `last_used` 最小的块——这和 MiniOS 第五阶段实现的 LRU 算法完全相同，只是现在用在了实际的缓存管理上。

### 检查点
- [ ] `block_read(100)` 第一次读磁盘，第二次从缓存返回（打印日志区分）
- [ ] 修改缓存数据后，`block_flush()` 能把 dirty 块写回磁盘
- [ ] 访问超过 64 个不同块时，LRU 正确淘汰最久未用的块

## 实验三：inode 文件系统，移植 MiniOS 第五阶段

### 目标
在磁盘上实现 inode 文件系统，文件和目录信息持久化存储。

### 背景知识：文件系统布局

将虚拟磁盘（`hdd.img`）划分成区域：

```
扇区 0：超级块（Superblock）
  → 记录文件系统的基本信息：总扇区数、inode 总数、位图起始位置等

扇区 1～4：inode 位图
  → 每个 bit 代表一个 inode 是否被使用（总共 4*512*8 = 16384 个 inode 槽）

扇区 5～8：数据块位图
  → 每个 bit 代表一个数据块是否被使用

扇区 9～264：inode 表
  → 每个 inode 占 64 字节，9到264扇区能存 256*512/64 = 2048 个 inode

扇区 265 以后：数据块
  → 实际存储文件内容和目录条目
```

**inode 结构（和 MiniOS 完全对应）：**

```c
typedef struct {
    uint32_t size;           // 文件大小（字节）
    uint8_t  type;           // 0 = 文件，1 = 目录
    uint32_t blocks[12];     // 直接块指针（12个数据块扇区号）
    uint32_t created_time;   // 创建时间（用 tick 计数代替真实时间戳）
} inode_t;                   // 大小要 <= 64 字节
```

**目录条目（Directory Entry）：**

```c
typedef struct {
    char     name[28];   // 文件名（最长27个字符）
    uint32_t inode_num;  // 对应的 inode 号（0 表示空条目）
} dir_entry_t;           // 每条 32 字节，一个扇区能放 16 条
```

### 步骤

**1. 新建 `filesystem/inode.h`**

```c
#ifndef INODE_H
#define INODE_H

#include <stdint.h>

#define MAX_INODES     2048
#define MAX_FILE_BLOCKS 12
#define MAX_FILENAME   27

typedef struct {
    uint32_t size;
    uint8_t  type;           // FS_TYPE_FILE = 0, FS_TYPE_DIR = 1
    uint32_t direct[12];     // 直接数据块地址（扇区号）
    uint32_t created_time;
    uint8_t  padding[15];    // 填充到 64 字节
} __attribute__((packed)) inode_t;

typedef struct {
    char     name[28];
    uint32_t inode_num;
} __attribute__((packed)) dir_entry_t;

void     inode_init();
int      inode_alloc();                      // 分配空闲 inode，返回 inode 号
void     inode_free(int inum);               // 释放 inode
inode_t* inode_get(int inum);               // 读取 inode（从块缓存）
void     inode_write(int inum, inode_t* node); // 写回 inode（到块缓存）

#endif
```

**2. 新建 `filesystem/fs.h`**

```c
#ifndef FS_H
#define FS_H

#include "inode.h"

// 文件系统格式化（首次运行时调用）
void fs_format();

// 文件系统挂载（非首次运行，读取超级块）
int  fs_mount();

// 文件操作（接口和 MiniOS 相同，但真正读写磁盘）
int  fs_mkdir(const char* path);
int  fs_create(const char* path);
int  fs_write(const char* path, const uint8_t* data, uint32_t size);
int  fs_read (const char* path, uint8_t* buffer, uint32_t size);
int  fs_ls   (const char* path);    // 列出目录内容（打印到 VGA）
void fs_stat (const char* path);    // 打印文件信息

#endif
```

**3. 实现 `filesystem/fs.c`（核心：fs_write 和 fs_read）**

**`fs_write`：**
```
1. 根据路径找到文件的 inode 号（遍历路径的每一层目录）
2. 分配数据块（调用 pmm_alloc_frame 或专用的块分配函数）
3. 把数据写入数据块（通过 block_write → 块缓存 → 最终到磁盘）
4. 更新 inode 的 direct[] 指针和 size 字段
5. 调用 inode_write 保存 inode
6. 调用 block_flush 确保写入磁盘
```

**`fs_read`：**
```
1. 根据路径找到 inode
2. 从 inode.direct[] 获取数据块地址列表
3. 逐块调用 block_read，拷贝数据到 buffer
4. 返回实际读取的字节数
```

**路径解析（fs_lookup）：**
```
输入："/dir1/file.txt"
步骤：
  从根目录 inode（inode 号 0）开始
  拆分路径：["dir1", "file.txt"]
  在当前目录的数据块里查找名为 "dir1" 的 dir_entry，得到其 inode 号
  以该 inode 号加载下一个目录
  重复，直到路径最后一段，返回目标 inode 号
```

**4. 在 `kernel_main` 里演示完整文件操作**

```c
// 第一次运行：格式化
fs_format();

// 创建目录和文件
fs_mkdir("/home");
fs_mkdir("/home/user");
fs_create("/home/user/hello.txt");

// 写入内容
uint8_t content[] = "Hello from TinyOS filesystem!\n";
fs_write("/home/user/hello.txt", content, sizeof(content));

// 读取内容
uint8_t buf[512];
int n = fs_read("/home/user/hello.txt", buf, sizeof(buf));
vga_puts("Read: ");
vga_puts((char*)buf);

// 列出目录
fs_ls("/home/user");
```

**预期输出：**
```
Read: Hello from TinyOS filesystem!
/home/user:
  hello.txt  [file, 31 bytes]
```

### 检查点
- [ ] `fs_format()` 后磁盘有有效的超级块和 inode 位图
- [ ] 创建文件、写入、读取内容正确（读出来和写入的一样）
- [ ] `mkdir` 创建目录，`ls` 能列出内容
- [ ] 重启 QEMU 后（不重新 format），数据依然存在（持久化验证）

## 实验四：与进程结合——进程写文件

### 目标
让第四阶段的进程能调用文件系统 API，每个进程把自己的输出写到不同的文件里。

### 步骤

**1. 修改进程函数**

```c
void process_a() {
    int count = 0;
    while (count < 5) {
        uint8_t msg[32];
        // 拼接消息（不能用 sprintf，手写一个简单版本）
        msg[0] = 'A'; msg[1] = ':'; msg[2] = '0' + count; msg[3] = '\n'; msg[4] = 0;
        
        fs_write("/proc_a.log", msg, 4);
        vga_putchar('A');
        count++;
        
        // 主动睡眠（通过多次 hlt 让出 CPU）
        for (volatile int i = 0; i < 5000000; i++);
    }
}

void process_b() {
    // 类似，写 /proc_b.log
}
```

**2. 验证：两个进程并发写文件，互不干扰**

文件系统操作是否需要加锁？这是一个开放问题：
- 简单做法：文件系统是内核态，同一时刻只有一个进程在内核态（单核）
- 复杂做法：用信号量保护 `block_write`（这就是 MiniOS 第四阶段信号量的用武之地！）

### 检查点
- [ ] 两个进程分别写入不同文件，内容互不混乱
- [ ] 进程结束后，用 `fs_read` 读取日志文件，内容正确

## 阶段总结

**本阶段把所有之前的模块连接在了一起：**

```
进程（第四阶段）
    ↓ 调用
文件系统 API（fs.c）
    ↓ 通过
inode 层（inode.c）→ 与 MiniOS filesystem.cpp 算法相同
    ↓ 通过
块缓存（block.c）→ 使用 MiniOS LRU 算法管理
    ↓ 通过
ATA 驱动（ata.c）→ 真正的磁盘读写
    ↓
QEMU 虚拟磁盘（hdd.img）
```

**与 MiniOS 的对应：**

| MiniOS | TinyOS | 联系 |
|--------|--------|------|
| `FileSystem::mkdir` | `fs.c::fs_mkdir` | 算法相同，但真正写磁盘 |

| SCAN 算法（磁盘调度） | **（本阶段未实现）** | 可选扩展 |

## 常见问题

**Q：写入文件后立刻读，读到的是旧数据？**  
A：写入通过块缓存（dirty 标记），读取也通过块缓存——如果缓存里有这个块，读的是缓存里已更新的数据，不会读到旧数据。但如果 QEMU 重启前没有 `block_flush()`，写入的数据可能没真正到磁盘。

**Q：路径解析时找不到文件，但我明明创建过？**  
A：检查路径是否从根目录 `/` 开始（根目录的 inode 固定为 0）。目录条目的查找是大小写敏感的，`/Home` 和 `/home` 是不同路径。

**Q：ATA 读写挂起（等待 BSY 或 DRQ 永不返回）？**  
A：确认 QEMU 命令里有 `-hda hdd.img`（连接了虚拟硬盘）。如果没有连接磁盘，`inb(0x1F7)` 会返回 0xFF，导致无限等待。加超时：等待超过 100000 次循环还未就绪则返回错误码。
