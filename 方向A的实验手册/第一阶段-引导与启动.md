# 实验手册 · TinyOS 第一阶段：引导，让 QEMU 启动你的代码

> **阶段目标：** QEMU 启动后，屏幕上打印 `Hello, TinyOS!`，证明你的代码已经在裸机上运行。  
> **预计时间：** 1～2 周，每天约 1 小时  
> **前置条件：** MiniOS 六个阶段全部完成  
> **本阶段涉及文件：**
> ```
> TinyOS/
> ├── Makefile
> ├── linker.ld
> ├── boot/
> │   └── boot.asm          ← 本阶段核心，512字节引导扇区
> ├── kernel/
> │   ├── kernel_entry.asm  ← 汇编跳转到C的桥梁
> │   └── kernel.c          ← C语言内核入口
> ├── drivers/
> │   └── vga.h/c           ← 直接写显存打印字符
> └── lib/
>     └── string.h/c        ← 自己实现 memset/memcpy（不能用标准库）
> ```
> **本阶段不涉及：** 中断、进程、内存分配——这些都留给后续阶段。

## 实验零：安装工具链

### 目标
安装 QEMU、NASM、交叉编译 GCC，验证环境正常。

### 背景知识：为什么需要交叉编译？

你的 Mac 是 64 位 ARM 或 x86_64 架构，但 TinyOS 要运行在 32 位 x86（i386）上。普通的 `gcc` 编译出来的程序依赖操作系统（它会链接 `libc`、调用系统调用），放在裸机上根本无法运行。

**交叉编译器（i686-elf-gcc）** 能编译出不依赖任何操作系统的纯二进制机器码，这才是内核需要的。

### 步骤

**macOS 安装：**

```bash
# 安装 QEMU（硬件模拟器）
brew install qemu

# 安装 NASM（汇编器）
brew install nasm

# 安装 i686-elf 交叉编译工具链
brew install i686-elf-gcc i686-elf-binutils
```

**验证安装：**

```bash
qemu-system-i386 --version   # 应该输出版本号
nasm --version                # 应该输出版本号
i686-elf-gcc --version        # 应该输出版本号
```

### 检查点
- [ ] 三个命令都能输出版本号
- [ ] `which i686-elf-gcc` 能找到路径

## 实验一：理解引导扇区，写第一行汇编

### 目标
写一个 512 字节的引导扇区，QEMU 启动时能加载并运行它（暂时什么都不做，只是能启动不崩溃）。

### 背景知识：BIOS 如何启动 OS

计算机上电后，第一个运行的程序不是 OS，而是主板上的 **BIOS**（基本输入输出系统）。BIOS 做完硬件自检后，会：

```
1. 读取磁盘第一个扇区（512字节，称为引导扇区 / MBR）
2. 把这 512 字节加载到内存地址 0x7C00
3. 跳转到 0x7C00 开始执行
```

BIOS 只有一个判断：**引导扇区的最后两个字节必须是 0x55、0xAA**（引导签名），否则认为这块磁盘不可引导，停止。

所以最简单的引导扇区只需要：
- 填充到 512 字节
- 最后两字节是 0x55AA
- 中间放你想执行的代码

**实模式（Real Mode）：**

BIOS 把控制权交给引导扇区时，CPU 处于 **实模式**（16位模式）：
- 只能访问 1MB 以内的内存
- 没有内存保护，任何代码能访问任何地址
- 段寄存器 + 偏移寻址（`CS:IP`、`DS:SI` 等）
- 可以使用 BIOS 中断（`int 0x10` 打印字符，`int 0x13` 读磁盘）

我们在这一阶段先在实模式里打印字符，然后再切换到保护模式。

### 步骤

**1. 创建项目目录**

```
TinyOS/
├── Makefile
├── boot/
│   └── boot.asm
└── kernel/
    └── kernel.c
```

**2. 编写 `boot/boot.asm`（第一版，什么都不做）**

NASM 汇编语法，需要理解以下指令：

| 指令 | 含义 |
|------|------|
| `[bits 16]` | 告诉 NASM：生成 16 位（实模式）代码 |
| `[org 0x7C00]` | 告诉 NASM：这段代码会被加载到 0x7C00，计算地址时以此为基准 |
| `jmp $` | 无限循环（`$` 表示当前地址，跳到自己 = 死循环） |
| `times 510-($-$$) db 0` | 用 0 填充到第 510 字节（`$$` 表示段起始地址） |
| `dw 0xAA55` | 最后两字节写入引导签名（注意小端序，0x55AA 写成 `0xAA55`） |

```nasm
[bits 16]
[org 0x7C00]

start:
    jmp $              ; 无限循环，暂时什么都不做

times 510-($-$$) db 0  ; 填充到 510 字节
dw 0xAA55              ; 引导签名
```

**3. 编写 `Makefile`**

```makefile
# 工具定义
NASM    = nasm
QEMU    = qemu-system-i386

# 构建引导扇区镜像
boot/boot.bin: boot/boot.asm
    $(NASM) -f bin boot/boot.asm -o boot/boot.bin

# 创建磁盘镜像
tinyos.img: boot/boot.bin
    dd if=/dev/zero of=tinyos.img bs=512 count=2880   # 创建 1.44MB 空白软盘镜像
    dd if=boot/boot.bin of=tinyos.img conv=notrunc    # 写入引导扇区

# 运行
run: tinyos.img
    $(QEMU) -drive format=raw,if=floppy,file=tinyos.img

clean:
    rm -f boot/boot.bin tinyos.img
```

**4. 编译并运行**

```bash
make
make run
```

QEMU 窗口应该打开，显示黑屏（没有崩溃报错就是成功）。

### 检查点
- [ ] `nasm -f bin boot/boot.asm -o boot/boot.bin` 没有报错
- [ ] `boot.bin` 文件大小恰好是 512 字节（`ls -l boot/boot.bin`）
- [ ] `make run` 后 QEMU 窗口打开，不崩溃（黑屏是正常的）

## 实验二：用 BIOS 中断打印字符（实模式）

### 目标
在实模式下，用 BIOS 提供的 `int 0x10` 中断打印 `Hello, TinyOS!`。

### 背景知识：BIOS 中断

实模式下，BIOS 提供了一组"系统服务"，通过软中断调用：

- `int 0x10`：视频服务（打印字符、设置光标等）
- `int 0x13`：磁盘服务（读写扇区）
- `int 0x16`：键盘服务

**打印单个字符（`int 0x10`，功能 0x0E）：**

```
AH = 0x0E    （功能号：Teletype Output）
AL = 字符的 ASCII 码
BH = 页号（通常填 0）
BL = 颜色（文本模式，通常填 0x07 = 白色）
执行 int 0x10
```

### 步骤

**1. 修改 `boot/boot.asm`**

```nasm
[bits 16]
[org 0x7C00]

start:
    ; 初始化段寄存器（实模式下必须做）
    xor ax, ax
    mov ds, ax
    mov es, ax
    mov ss, ax
    mov sp, 0x7C00    ; 栈从 0x7C00 向下增长

    ; 打印字符串
    mov si, msg       ; SI 指向字符串首地址
.loop:
    lodsb             ; 从 DS:SI 加载一个字节到 AL，SI 自动 +1
    or al, al         ; 检查是否为 0（字符串结束符）
    jz .done          ; 如果是 0，跳到结束
    mov ah, 0x0E      ; BIOS 功能：打印字符
    mov bh, 0         ; 页号
    int 0x10          ; 调用 BIOS
    jmp .loop

.done:
    jmp $             ; 无限循环

msg db 'Hello, TinyOS!', 0x0D, 0x0A, 0   ; 字符串，\r\n，结束符0

times 510-($-$$) db 0
dw 0xAA55
```

**2. 编译运行**

```bash
make clean && make && make run
```

QEMU 窗口中应该出现 `Hello, TinyOS!`。

**3. 理解每一行**

对照注释，逐行读懂代码，重点理解：
- `xor ax, ax` 为什么比 `mov ax, 0` 更常见（等价但少一个字节）
- `lodsb` 指令做了什么（加载 + 自动递增）
- 为什么字符串末尾要有 `0`（C 风格字符串结束符）

### 检查点
- [ ] QEMU 屏幕上出现 `Hello, TinyOS!`
- [ ] 能解释 `lodsb` 指令的作用
- [ ] 能解释为什么要初始化段寄存器

## 实验三：加载内核，从磁盘读取更多代码

### 目标
引导扇区只有 512 字节，不够放内核。用 BIOS `int 0x13` 从磁盘第二个扇区读取内核代码，然后跳转过去执行。

### 背景知识：磁盘读取

BIOS `int 0x13` 功能 0x02：

```
AH = 0x02       （功能：读扇区）
AL = 要读的扇区数
CH = 柱面号（通常0）
CL = 扇区号（从1开始！不是从0）
DH = 磁头号（通常0）
DL = 驱动器号（0x00=软盘，0x80=硬盘）
ES:BX = 目标内存地址（读到哪里）
执行 int 0x13
```

内核将被读到内存 `0x1000:0x0000`（物理地址 0x10000 = 64KB处）。

### 步骤

**1. 修改 `boot/boot.asm`，加入磁盘读取**

```nasm
[bits 16]
[org 0x7C00]

start:
    xor ax, ax
    mov ds, ax
    mov es, ax
    mov ss, ax
    mov sp, 0x7C00

    ; 打印 "Loading..."
    mov si, loading_msg
    call print_string

    ; 读取内核（从第2扇区开始，读取5个扇区到 0x1000:0x0000）
    mov ax, 0x1000
    mov es, ax          ; 目标段地址
    mov bx, 0           ; 目标偏移
    mov ah, 0x02        ; 功能：读扇区
    mov al, 5           ; 读5个扇区
    mov ch, 0           ; 柱面0
    mov cl, 2           ; 从第2扇区开始（第1扇区是引导扇区）
    mov dh, 0           ; 磁头0
    mov dl, 0           ; 软盘驱动器0
    int 0x13
    jc disk_error       ; CF=1 表示读取失败

    ; 跳转到内核
    jmp 0x1000:0x0000

disk_error:
    mov si, error_msg
    call print_string
    jmp $

print_string:
    lodsb
    or al, al
    jz .done
    mov ah, 0x0E
    mov bh, 0
    int 0x10
    jmp print_string
.done:
    ret

loading_msg db 'Loading TinyOS...', 0x0D, 0x0A, 0
error_msg   db 'Disk read error!', 0

times 510-($-$$) db 0
dw 0xAA55
```

**2. 新建 `kernel/kernel_entry.asm`（最简版）**

```nasm
[bits 16]
[org 0x0000]       ; 这段代码会被加载到 0x1000:0x0000

start:
    mov si, msg
    call print_string
    jmp $

print_string:
    lodsb
    or al, al
    jz .done
    mov ah, 0x0E
    xor bh, bh
    int 0x10
    jmp print_string
.done:
    ret

msg db 'Kernel loaded!', 0x0D, 0x0A, 0

times 512*5-($-$$) db 0   ; 填充到5个扇区大小
```

**3. 更新 Makefile**

```makefile
kernel/kernel_entry.bin: kernel/kernel_entry.asm
    $(NASM) -f bin kernel/kernel_entry.asm -o kernel/kernel_entry.bin

tinyos.img: boot/boot.bin kernel/kernel_entry.bin
    dd if=/dev/zero of=tinyos.img bs=512 count=2880
    dd if=boot/boot.bin of=tinyos.img conv=notrunc
    dd if=kernel/kernel_entry.bin of=tinyos.img bs=512 seek=1 conv=notrunc
    # seek=1 表示从第2个扇区开始写（跳过引导扇区）
```

**4. 运行，看到两行输出**

```
Loading TinyOS...
Kernel loaded!
```

### 检查点
- [ ] 引导扇区能成功读取磁盘第 2 扇区
- [ ] 跳转到内核代码后，能打印 `Kernel loaded!`
- [ ] 理解 `jc disk_error` 的含义（CF 标志位，Carry Flag）

## 实验四：切换保护模式，接入 C 内核

### 目标
从 16 位实模式切换到 32 位保护模式，然后跳转到用 C 写的内核函数，此后所有代码都用 C 写。

### 背景知识：为什么要切换保护模式？

实模式的限制：
- 只能访问 1MB 内存
- 没有内存保护（任何代码能改任何内存）
- 16 位寄存器，效率低

保护模式（32 位）：
- 最多访问 4GB 内存
- 内存保护（用户态不能访问内核内存）
- 32 位寄存器，运行效率高
- 支持虚拟内存（分页）

**切换步骤（固定套路）：**
```
1. 关闭中断（cli）
2. 加载 GDT（全局描述符表）
3. 设置 CR0 寄存器的第 0 位为 1（开启保护模式）
4. 远跳转（far jump）刷新指令流水线，切换到 32 位代码段
5. 重置所有段寄存器（ds、ss、es 等）
6. 设置栈指针，跳转到 C 函数
```

**GDT（全局描述符表）：**

保护模式下，内存访问通过"段描述符"来定义每段内存的基址、大小、权限。GDT 是这些描述符的表格。最简单的 GDT 包含：
- 第 0 项：空描述符（必须，CPU 规定）
- 第 1 项：代码段（可读可执行，覆盖 0～4GB）
- 第 2 项：数据段（可读可写，覆盖 0～4GB）

这种"覆盖整个 4GB"的简化布局叫做**平坦内存模型（Flat Memory Model）**，是最简单的配置。

### 步骤

**1. 重写 `boot/boot.asm`，加入 GDT 和模式切换**

```nasm
[bits 16]
[org 0x7C00]

start:
    ; 关闭中断，初始化段寄存器
    cli
    xor ax, ax
    mov ds, ax
    mov es, ax
    mov ss, ax
    mov sp, 0x7C00

    ; 打印提示
    sti
    mov si, msg_loading
    call print16
    cli

    ; 从磁盘加载内核到 0x10000
    ; ...（同实验三，省略）

    ; 加载 GDT
    lgdt [gdt_descriptor]

    ; 开启保护模式（设置 CR0 第0位）
    mov eax, cr0
    or eax, 1
    mov cr0, eax

    ; 远跳转到 32 位代码（刷新流水线，切换到保护模式代码段）
    ; 0x08 是代码段选择子（GDT 第1项，每项8字节，所以偏移=8=0x08）
    jmp 0x08:protected_mode_start

; ── GDT 定义 ──────────────────────────────────────────────────
gdt_start:
    ; 第0项：空描述符（8个0字节）
    dq 0

    ; 第1项：代码段（基址=0，限长=4GB，可执行可读，DPL=0）
    ; 格式复杂，按位拼接，直接用这个固定值：
    dw 0xFFFF    ; 段限长低16位
    dw 0x0000    ; 基址低16位
    db 0x00      ; 基址中8位
    db 10011010b ; 访问权限（代码段，Ring 0，可读可执行）
    db 11001111b ; 标志 + 段限长高4位（4KB粒度，32位，限长=0xFFFFF）
    db 0x00      ; 基址高8位

    ; 第2项：数据段（基址=0，限长=4GB，可读可写，DPL=0）
    dw 0xFFFF
    dw 0x0000
    db 0x00
    db 10010010b ; 访问权限（数据段，Ring 0，可读可写）
    db 11001111b
    db 0x00
gdt_end:

gdt_descriptor:
    dw gdt_end - gdt_start - 1   ; GDT 大小（字节数-1）
    dd gdt_start                  ; GDT 物理地址

; ── 32位保护模式代码 ────────────────────────────────────────────
[bits 32]
protected_mode_start:
    ; 重置数据段（选择子 0x10 = GDT 第2项）
    mov ax, 0x10
    mov ds, ax
    mov ss, ax
    mov es, ax
    mov fs, ax
    mov gs, ax

    ; 设置栈（在 1MB 以下的安全位置）
    mov esp, 0x90000

    ; 跳转到 C 内核（地址 0x10000）
    call 0x10000

    jmp $

; ── 16位辅助函数 ────────────────────────────────────────────────
[bits 16]
print16:
    lodsb
    or al, al
    jz .done
    mov ah, 0x0E
    xor bh, bh
    int 0x10
    jmp print16
.done:
    ret

msg_loading db 'Switching to protected mode...', 0x0D, 0x0A, 0

times 510-($-$$) db 0
dw 0xAA55
```

**2. 新建 `kernel/kernel.c`，这是第一个 C 内核函数**

```c
// 不能 #include 任何标准库！
// 直接操作 VGA 显存（下一个实验实现完整的 VGA 驱动）

void kernel_main() {
    // VGA 文字模式显存起始地址
    char* vga = (char*)0xB8000;
    
    // 写入 'O'（字符 + 颜色属性，每个字符占2字节）
    vga[0] = 'O';   // 字符
    vga[1] = 0x0F;  // 颜色：黑底白字
    vga[2] = 'K';
    vga[3] = 0x0F;
    
    while (1) {}    // 无限循环
}
```

**3. 新建 `kernel/kernel_entry.asm`（正式版，调用 C 函数）**

```nasm
[bits 32]
[extern kernel_main]   ; 声明 kernel_main 是外部 C 函数

global _start
_start:
    call kernel_main
    jmp $
```

**4. 新建 `linker.ld`**

```ld
ENTRY(_start)

SECTIONS {
    . = 0x10000;        /* 内核加载到物理地址 0x10000 */
    
    .text : {
        *(.text)        /* 所有代码段 */
    }
    
    .rodata : {
        *(.rodata)      /* 只读数据（字符串常量等） */
    }
    
    .data : {
        *(.data)        /* 已初始化数据 */
    }
    
    .bss : {
        *(.bss)         /* 未初始化数据（清零） */
    }
}
```

**5. 更新 Makefile，用 GCC 编译 C 内核**

```makefile
CC      = i686-elf-gcc
LD      = i686-elf-ld
NASM    = nasm
QEMU    = qemu-system-i386

# GCC 编译选项（裸机，不链接任何库）
CFLAGS  = -m32 -ffreestanding -fno-stack-protector -nostdlib -nostdinc -O2

all: tinyos.img

boot/boot.bin: boot/boot.asm
    $(NASM) -f bin $< -o $@

kernel/kernel_entry.o: kernel/kernel_entry.asm
    $(NASM) -f elf32 $< -o $@

kernel/kernel.o: kernel/kernel.c
    $(CC) $(CFLAGS) -c $< -o $@

drivers/vga.o: drivers/vga.c
    $(CC) $(CFLAGS) -c $< -o $@

kernel/kernel.elf: kernel/kernel_entry.o kernel/kernel.o drivers/vga.o
    $(LD) -T linker.ld -o $@ $^

kernel/kernel.bin: kernel/kernel.elf
    i686-elf-objcopy -O binary $< $@

tinyos.img: boot/boot.bin kernel/kernel.bin
    dd if=/dev/zero of=$@ bs=512 count=2880
    dd if=boot/boot.bin of=$@ conv=notrunc
    dd if=kernel/kernel.bin of=$@ bs=512 seek=1 conv=notrunc

run: tinyos.img
    $(QEMU) -drive format=raw,if=floppy,file=$<

clean:
    rm -f boot/*.bin kernel/*.o kernel/*.bin kernel/*.elf drivers/*.o tinyos.img
```

**6. 编译运行，屏幕上出现 `OK`**

> **注意：** `i686-elf-ld` 默认输出 ELF 格式文件（含文件头和段表元数据），不能直接用 `dd` 写入磁盘镜像。必须用 `i686-elf-objcopy -O binary` 将 ELF 转换为纯二进制，否则引导扇区跳转后会执行 ELF header 而非内核代码，导致 Triple Fault（CPU 不断重置重启）。

### 检查点
- [ ] 成功切换到保护模式（不崩溃）
- [ ] C 函数 `kernel_main` 被调用
- [ ] VGA 显存写入后，屏幕左上角出现 `OK`（屏幕有残留文字是正常的，此阶段未实现清屏）
- [ ] 能解释：为什么 `jmp 0x08:protected_mode_start` 是一个"远跳转"，它做了什么？
- [ ] 能解释：为什么需要 `objcopy` 将 ELF 转为纯二进制？

## 实验五：实现 VGA 驱动，能打印字符串

### 目标
把直接写显存的操作封装成 `kprintf` 函数，内核从此可以方便地打印信息。

### 背景知识：VGA 文字模式

VGA 文字模式下，显存从 `0xB8000` 开始，每个字符占 **2 个字节**：

```
字节 0：ASCII 字符码
字节 1：颜色属性
  高 4 位 = 背景色
  低 4 位 = 前景色（字符颜色）
  
常用颜色：
  0x0F = 黑底白字
  0x0A = 黑底绿字
  0x0C = 黑底红字
```

文字模式默认 80 列 × 25 行，字符 `(col, row)` 在显存的位置：

```c
int offset = (row * 80 + col) * 2;
```

### 步骤

**1. 新建 `drivers/vga.h`**

```c
#ifndef VGA_H
#define VGA_H

// 颜色定义
#define VGA_COLOR_BLACK   0
#define VGA_COLOR_GREEN   2
#define VGA_COLOR_RED     4
#define VGA_COLOR_WHITE   15

// 组合前景色和背景色
#define VGA_COLOR(fg, bg) ((bg << 4) | fg)

void vga_init();
void vga_putchar(char c);
void vga_puts(const char* str);
void vga_clear();

#endif
```

**2. 新建 `drivers/vga.c`**

需要实现：

- `vga_clear()`：把显存全部填为空格（`' '`，颜色 `0x0F`）
- `vga_putchar(char c)`：
  - 普通字符：在当前光标位置写入，光标向右移动
  - `'\n'`：光标移到下一行行首
  - 到达底部时：向上滚动一行（把第 1～24 行的内容复制到第 0～23 行，清空第 24 行）
- `vga_puts(const char* str)`：循环调用 `vga_putchar`

**注意：** 不能用 `printf`、`strlen` 等标准库函数，需要手写。

**3. 新建 `lib/string.h` 和 `lib/string.c`**

实现裸机版的基础字符串函数：

```c
void*  memset(void* ptr, int value, int size);
void*  memcpy(void* dst, const void* src, int size);
int    strlen(const char* str);
```

这些函数在 MiniOS 里用的是标准库，现在需要自己实现。

**4. 修改 `kernel/kernel.c`**

```c
#include "../drivers/vga.h"

void kernel_main() {
    vga_init();
    vga_clear();
    vga_puts("Hello, TinyOS!\n");
    vga_puts("Kernel is running in 32-bit protected mode.\n");
    
    while (1) {}
}
```

### 检查点
- [ ] 屏幕上出现 `Hello, TinyOS!`（绿色或白色字体，黑色背景）
- [ ] `vga_putchar('\n')` 能正确换行
- [ ] 打印超过 25 行时，内容能向上滚动
- [ ] `memset`、`memcpy`、`strlen` 自己实现，编译通过

## 常见问题

**Q：QEMU 启动后一片黑，什么都没有？**  
A：按以下顺序排查：
1. `boot.bin` 是否恰好 512 字节（`wc -c boot/boot.bin`）
2. 最后两字节是否是 `55 AA`（`xxd boot/boot.bin | tail -1`）
3. `dd` 命令是否真的把 `boot.bin` 写到了 `tinyos.img` 的第一个扇区

**Q：切换保护模式后 QEMU 重启/三重错误？**  
A：最常见原因：
1. GDT 描述符格式写错（某个位填错了）
2. 远跳转的段选择子不对（应该是 0x08，不是 0x01 或 1）
3. 切换后没有重新设置 `ds`、`ss` 等段寄存器

**Q：C 代码里能用 `=`、`+` 等运算符，但不能用 `printf`，为什么？**  
A：运算符是 CPU 指令，不依赖操作系统。`printf` 是 C 标准库函数，标准库内部调用了操作系统的系统调用（`write`）。裸机上没有操作系统，所以不能用任何依赖系统调用的库函数。
