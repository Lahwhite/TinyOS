# 实验手册 · TinyOS 第四阶段：进程，真正的多任务切换

> **阶段目标：** 创建 3 个进程，时钟中断触发调度器切换，屏幕上能看到进程交替打印字符——这是 TinyOS 和 MiniOS 最大的区别。  
> **预计时间：** 2 周，每天约 1 小时  
> **前置条件：** 第三阶段完成，`kmalloc` 可用，时钟中断正常工作  
> **本阶段新增文件：**
> ```
> TinyOS/
> └── process/
>     ├── process.h/c   ← PCB（从 MiniOS 移植，去掉 STL）
>     └── scheduler.h/c ← 调度器（算法从 MiniOS 移植，接入时钟中断）
> ```

---

## 实验一：理解上下文切换

### 背景知识：MiniOS 和 TinyOS 的调度差距

**MiniOS 的调度（模拟）：**
```cpp
scheduler.run("RR", mem);   // 你手动调用这个函数
// 内部：for 循环，每次 dispatch() 然后 terminate()
// 进程"运行"只是修改了 state 字段，没有真正执行任何代码
```

**TinyOS 的调度（真实）：**
```
进程 A 的代码在 CPU 上执行
    ↓
时钟中断触发（每 10ms）
    ↓
CPU 自动保存 EIP、EFLAGS、CS 到栈
    ↓
isr.asm 里 pusha 保存所有通用寄存器
    ↓
调度器决定：下一个运行 B
    ↓
把 B 的寄存器快照恢复到 CPU
    ↓
iret 返回，CPU 跳到 B 保存的 EIP 继续执行
    ↓
进程 B 完全不知道发生过中断，继续它的代码
```

**上下文（Context）** 就是 CPU 在某一时刻的完整状态：
- 通用寄存器（EAX/EBX/ECX/EDX/ESI/EDI/EBP/ESP）
- 段寄存器（CS/DS/SS/ES/FS/GS）
- EIP（下一条要执行的指令地址）
- EFLAGS（标志寄存器：进位、零、溢出等）

进程切换 = 保存当前进程的上下文 + 恢复目标进程的上下文。

## 实验二：定义 PCB，移植自 MiniOS

### 目标
定义不依赖任何 STL 的 PCB 结构体，包含上下文保存所需的字段。

### 步骤

**1. 新建 `process/process.h`**

和 MiniOS 的 PCB 字段基本一致，但数据类型全部换成 C 的：

```c
#ifndef PROCESS_H
#define PROCESS_H

#include <stdint.h>

// 进程状态
typedef enum {
    PROCESS_NEW        = 0,
    PROCESS_READY      = 1,
    PROCESS_RUNNING    = 2,
    PROCESS_BLOCKED    = 3,
    PROCESS_TERMINATED = 4
} process_state_t;

// CPU 上下文（切换时保存/恢复的寄存器）
typedef struct {
    uint32_t edi, esi, ebp, esp;
    uint32_t ebx, edx, ecx, eax;
    uint32_t eip;        // 下一条指令地址
    uint32_t eflags;     // 标志寄存器
} cpu_context_t;

// PCB（进程控制块）
typedef struct process {
    int              pid;
    char             name[32];       // 固定长度，不用 std::string
    process_state_t  state;
    int              priority;
    int              burst_time;
    int              remaining_time;
    
    cpu_context_t    context;        // 上下文（MiniOS 里没有，这是新增的核心字段）
    uint32_t*        stack;          // 进程栈的起始地址（kmalloc 分配）
    uint32_t         stack_size;     // 栈大小

    struct process*  next;           // 链表指针（就绪队列用链表，不用 std::vector）
} process_t;

// 函数声明
process_t* process_create(const char* name, int priority, void (*entry_func)());
void       process_destroy(process_t* p);

#endif
```

**注意与 MiniOS 的区别：**

| MiniOS PCB | TinyOS PCB | 原因 |
|-----------|-----------|------|
| `std::string name` | `char name[32]` | 没有 STL |
| 无 `context` 字段 | `cpu_context_t context` | 真实切换需要保存寄存器 |
| 无 `stack` 字段 | `uint32_t* stack` | 每个进程需要独立的栈 |
| `next_inode_num` 无 | `struct process* next` | 就绪队列用手写链表 |

**2. 新建 `process/process.c`**

实现 `process_create`：

```c
process_t* process_create(const char* name, int priority, void (*entry_func)()) {
    process_t* proc = (process_t*)kmalloc(sizeof(process_t));
    
    // 分配进程栈（8KB）
    proc->stack = (uint32_t*)kmalloc(8192);
    proc->stack_size = 8192;
    
    // 初始化上下文：让进程第一次被调度时跳转到 entry_func
    // 栈从高地址向低地址增长，初始 ESP 指向栈顶
    uint32_t* stack_top = proc->stack + (8192 / sizeof(uint32_t));
    
    // 把入口函数地址放到栈上，作为"返回地址"
    // 当调度器 iret 时，EIP = entry_func，进程开始执行
    *(--stack_top) = (uint32_t)entry_func;
    
    proc->context.esp    = (uint32_t)stack_top;
    proc->context.eip    = (uint32_t)entry_func;
    proc->context.eflags = 0x200;   // 中断使能位（IF=1）
    
    // 其他字段
    // strncpy(proc->name, name, 31);  // 需要自己实现 strncpy（lib/string.c）
    proc->priority      = priority;
    proc->state         = PROCESS_READY;
    proc->remaining_time = 10;       // 默认时间片
    proc->next          = NULL;
    
    return proc;
}
```

### 检查点
- [ ] `process_create` 能分配 PCB 和栈（依赖 `kmalloc`）
- [ ] PCB 结构体不包含任何 STL 类型
- [ ] 编译通过（`i686-elf-gcc -c process.c`）

## 实验三：实现调度器，移植 RR 算法

### 目标
实现就绪队列（手写链表）和 RR 调度器，调度器将被时钟中断调用。

### 步骤

**1. 新建 `process/scheduler.h`**

```c
#ifndef SCHEDULER_H
#define SCHEDULER_H

#include "process.h"

void scheduler_init();
void scheduler_add(process_t* proc);   // 加入就绪队列
void scheduler_tick();                  // 时钟中断调用，决定是否切换进程
process_t* scheduler_current();         // 返回当前运行的进程

#endif
```

**2. 新建 `process/scheduler.c`**

内部数据：
```c
static process_t* ready_queue_head = NULL;  // 就绪队列头
static process_t* current_proc     = NULL;  // 当前运行进程
static int        time_quantum      = 5;     // 时间片（tick数）
```

实现就绪队列（环形链表，适合 RR）：
```c
void scheduler_add(process_t* proc) {
    proc->state = PROCESS_READY;
    if (ready_queue_head == NULL) {
        ready_queue_head = proc;
        proc->next = proc;   // 环形：指向自身
    } else {
        // 找到链表尾部，插入
        process_t* tail = ready_queue_head;
        while (tail->next != ready_queue_head) tail = tail->next;
        tail->next = proc;
        proc->next = ready_queue_head;
    }
}
```

实现 `scheduler_tick`（RR 逻辑，从 MiniOS 移植）：
```c
void scheduler_tick() {
    if (current_proc == NULL || ready_queue_head == NULL) return;
    
    current_proc->remaining_time--;
    
    if (current_proc->remaining_time <= 0) {
        // 时间片用完，切换到下一个进程
        current_proc->remaining_time = time_quantum;
        current_proc = current_proc->next;   // 环形队列，直接取 next
        // 实际的寄存器切换在汇编里做（下一个实验）
    }
}
```

### 检查点
- [ ] `scheduler_add` 能正确构建环形链表
- [ ] `scheduler_tick` 能按 RR 逻辑轮换 `current_proc`

## 实验四：实现真正的上下文切换（核心）

### 目标
在时钟中断处理函数里，把当前进程的寄存器保存到 PCB，再把下一个进程的寄存器恢复到 CPU，实现真正的进程切换。

### 背景知识：切换发生在哪里

```
时钟中断触发
    ↓
isr.asm::irq_common：
    pusha          ← 保存寄存器到【当前进程的内核栈】
    call irq_handler (C 函数)
        ↓
        scheduler_tick()
          ↓
          context_switch(old, new)   ← 核心！
            把 old 的寄存器快照保存到 old->context
            把 new->context 恢复到 CPU 寄存器
    ↓
    popa           ← 恢复寄存器（现在恢复的是新进程的寄存器）
    iret           ← 跳到新进程的 EIP 继续执行
```

**实现上下文切换的关键：** 修改中断处理返回时栈上的值。`iret` 从栈上读取 EIP/EFLAGS/CS，如果我们在 `irq_handler` 里修改了栈上这些位置的值，`iret` 就会跳到不同的地方。

### 步骤

**1. 修改 `kernel/isr.asm`，让 IRQ 处理时传递栈指针**

```nasm
irq_common:
    pusha
    push ds
    push es
    push fs
    push gs

    mov ax, 0x10
    mov ds, ax
    mov es, ax

    push esp            ; ← 新增：把当前 ESP 传给 C 函数
                        ;   C 函数拿到的是指向栈上寄存器快照的指针
    call irq_handler
    add esp, 4          ; 弹出 push esp 的参数

    pop gs
    pop fs
    pop es
    pop ds
    popa
    add esp, 8
    iret
```

**2. 修改 `kernel/isr.c` 里 IRQ 处理函数的签名**

```c
// 参数变成栈指针，可以通过它修改返回时的寄存器值
uint32_t irq_handler(uint32_t esp) {
    // ... 分发到各 IRQ 处理函数
    if (irq_num == 0) {           // IRQ0 = 时钟
        uint32_t new_esp = timer_interrupt(esp);
        pic_send_eoi(0);
        return new_esp;           // 返回新的 ESP（可能是另一个进程的栈）
    }
    pic_send_eoi(irq_num - 32);
    return esp;
}
```

**3. 在 `kernel/timer.c` 里实现真正的切换**

```c
uint32_t timer_interrupt(uint32_t current_esp) {
    ticks++;
    
    if (current_proc == NULL) return current_esp;
    
    // 保存当前进程的栈指针（栈上有所有寄存器的快照）
    current_proc->context.esp = current_esp;
    
    // 调用调度器，更新 current_proc
    scheduler_tick();
    
    // 返回新进程的栈指针（isr.asm 会用这个 ESP 恢复寄存器）
    return current_proc->context.esp;
}
```

**4. 理解为什么这样能切换进程**

```
进程 A 被中断时，栈上的内容（从栈顶到栈底）：
  [gs][fs][es][ds][edi][esi][ebp][esp][ebx][edx][ecx][eax]  ← pusha 压入
  [中断号][错误码]                                            ← isr.asm 压入
  [EIP][CS][EFLAGS]                                          ← CPU 自动压入

timer_interrupt 保存 current_esp（指向这个栈），更新 current_proc 为进程 B

isr.asm 用返回的 new_esp（进程 B 的栈指针）恢复寄存器
popa 弹出 B 的通用寄存器
iret 弹出 B 的 EIP/CS/EFLAGS

CPU 跳到 B 的 EIP，进程 B 开始执行！
```

### 检查点
- [ ] 创建 2 个进程（分别打印 'A' 和 'B'），时钟中断触发后能交替打印
- [ ] 进程 A 和 B 互不干扰，各自的寄存器状态正确保存/恢复
- [ ] 这是整个 TinyOS 最难的部分——如果出现三重错误，耐心用 QEMU 的 GDB 调试

## 实验五：三个进程，完整演示

### 目标
创建 3 个进程，在 `kernel_main` 里启动调度，观察进程在时钟中断驱动下交替运行。

### 步骤

**1. 在 `kernel.c` 里定义三个进程函数**

```c
void process_a() {
    while (1) {
        vga_putchar('A');
        // 主动让出 CPU（可选，让演示效果更明显）
        for (volatile int i = 0; i < 10000000; i++);
    }
}

void process_b() {
    while (1) {
        vga_putchar('B');
        for (volatile int i = 0; i < 10000000; i++);
    }
}

void process_c() {
    while (1) {
        vga_putchar('C');
        for (volatile int i = 0; i < 10000000; i++);
    }
}
```

**2. 在 `kernel_main` 里创建并启动**

```c
void kernel_main() {
    vga_init();
    vga_clear();
    idt_init();
    pic_init();
    pmm_init();
    vmm_init();
    heap_init(HEAP_START, HEAP_SIZE);
    
    scheduler_init();
    
    // 创建三个进程
    process_t* pa = process_create("proc_A", 1, process_a);
    process_t* pb = process_create("proc_B", 1, process_b);
    process_t* pc = process_create("proc_C", 1, process_c);
    scheduler_add(pa);
    scheduler_add(pb);
    scheduler_add(pc);
    
    // 设置第一个要运行的进程
    current_proc = pa;
    current_proc->state = PROCESS_RUNNING;
    
    // 启动时钟中断（此后调度器自动运行）
    timer_init(100);
    asm volatile("sti");
    
    // 内核主线程等待
    while (1) { asm volatile("hlt"); }
}
```

### 预期输出

屏幕上出现类似：
```
AAAAAABBBBBBBCCCCCCCAAAAAABBBBBBB...
```

每次时钟中断后切换，连续打印 5 个字符（一个时间片）左右。

### 检查点
- [ ] 三个进程都在运行，屏幕上能看到 A、B、C 交替出现
- [ ] 关闭某个进程（让它的函数返回），其他进程继续运行
- [ ] 理解：进程 A 执行时，B 和 C 在哪里？（在就绪队列里等待时钟中断）

## 阶段总结

**这是 TinyOS 最核心的阶段**，也是 MiniOS 做不到的事：

| 对比 | MiniOS | TinyOS |
|------|--------|--------|
| 调度触发方式 | 手动调用 `scheduler.run()` | 时钟中断强制触发 |
| 进程"运行" | 修改 state 字段 | 真正在 CPU 上执行代码 |
| 上下文切换 | 不存在 | 保存/恢复全部寄存器 |
| 进程间隔离 | 无（共享同一个栈） | 有（每个进程独立的栈） |

## 常见问题

**Q：创建进程后启动调度，立刻三重错误？**  
A：最常见原因是初始上下文设置错误。进程第一次被调度时，`iret` 会从栈上读 EIP/CS/EFLAGS，必须保证这三个值正确压栈。检查 `process_create` 里初始化栈的代码，特别是 CS 必须是内核代码段选择子（0x08）。

**Q：两个进程交替，但寄存器值乱了（计算结果错误）？**  
A：上下文保存/恢复有遗漏。确认所有通用寄存器（EAX/EBX/ECX/EDX/ESI/EDI/EBP/ESP）都被 `pusha`/`popa` 处理，并且 ESP 的切换发生在 `popa` 之前。

**Q：进程函数返回后系统崩溃？**  
A：进程函数不能直接 `return`（返回到哪里？栈上没有合法地址）。在进程函数末尾必须调用 `process_exit()` 或 `scheduler_remove(current_proc)` 来从就绪队列移除自己并触发调度切换。
