# 实验手册 · TinyOS 第三阶段：内存管理，内核能动态分配内存

> **阶段目标：** 实现 `kmalloc` / `kfree`，内核代码能动态分配内存；开启分页，虚拟地址和物理地址分离。  
> **预计时间：** 1～2 周，每天约 1 小时  
> **前置条件：** 第二阶段完成，中断和时钟正常工作  
> **本阶段新增文件：**
> ```
> TinyOS/
> └── memory/
>     ├── pmm.h/c     ← 物理内存管理（位图）
>     ├── vmm.h/c     ← 虚拟内存管理（页表）
>     └── heap.h/c    ← 内核堆（kmalloc/kfree）
> ```

---

## 实验一：探测物理内存，建立内存位图

### 目标
让内核知道系统有多少物理内存，哪些范围可以使用，建立位图记录页帧的使用情况。

### 背景知识：内核如何知道有多少内存？

BIOS 在启动时会把内存布局信息写到一个固定区域。在进入保护模式之前（实模式下），可以通过 `int 0x15, EAX=0xE820` 查询 BIOS 内存映射：

```
每次调用返回一条内存区间记录：
  基地址（64位）
  长度（64位）
  类型：1 = 可用，2 = 保留（不能用），3 = ACPI，...
```

**简化方案（适合学习）：** 直接假设前 32MB 内存可用，其中：
- `0x0000 ~ 0x00FF`：BIOS 中断向量表，不能用
- `0x7C00 ~ 0x7DFF`：引导扇区，可以回收
- `0x10000 ~ 内核结束`：内核代码和数据，不能用
- `内核结束 ~ 0x2000000`：可以自由分配

**页帧（Frame）：**

物理内存被切成等大的页帧，每帧 **4KB（4096字节）**。位图里每个 bit 代表一个页帧：

```
位图：[0][1][0][0][1][1]...
       ↑  ↑  ↑  ↑  ↑  ↑
     帧0 帧1 帧2 帧3 帧4 帧5
      空   占  空  空   占  占
```

32MB 内存 = 32 * 1024 / 4 = 8192 个页帧，位图大小 = 8192 / 8 = 1024 字节，非常小。

### 步骤

**1. 新建 `memory/pmm.h`**

```c
#ifndef PMM_H
#define PMM_H

#include <stdint.h>

#define PAGE_SIZE    4096
#define MEM_SIZE     (32 * 1024 * 1024)     // 假设 32MB 物理内存
#define TOTAL_FRAMES (MEM_SIZE / PAGE_SIZE)  // 8192 个页帧

void   pmm_init();
int    pmm_alloc_frame();    // 分配一个空闲页帧，返回帧号（失败返回-1）
void   pmm_free_frame(int frame_num);  // 释放页帧
void   pmm_mark_used(uint32_t phys_addr, uint32_t size);  // 标记一段物理内存为已用

#endif
```

**2. 新建 `memory/pmm.c`**

内部数据：
```c
static uint8_t bitmap[TOTAL_FRAMES / 8];  // 位图，1KB
```

实现 `pmm_init`：
```
// 初始化：全部标记为空闲
memset(bitmap, 0, sizeof(bitmap));

// 标记不能用的区域
pmm_mark_used(0x0000, 0x1000);    // BIOS 数据
pmm_mark_used(0x7C00, 0x200);     // 引导扇区
pmm_mark_used(KERNEL_START, KERNEL_SIZE);  // 内核本身
```

实现 `pmm_alloc_frame`：
```
遍历 bitmap，找第一个为 0 的 bit
将其设为 1（标记为已用）
返回对应的帧号
```

实现 `pmm_free_frame`：
```
将 bitmap 中对应 bit 清 0
```

**3. 在 `kernel_main` 里初始化并测试**

```c
pmm_init();
int f1 = pmm_alloc_frame();
int f2 = pmm_alloc_frame();
vga_puts("Allocated frames: ");
// 打印 f1 和 f2 的值（应该是连续的帧号）
pmm_free_frame(f1);
```

### 检查点
- [ ] `pmm_alloc_frame` 返回的帧号在内核区域之外
- [ ] 连续分配两个帧，帧号连续
- [ ] 释放后再分配，能重用已释放的帧号
- [ ] 分配 8192 个帧后返回 -1（内存耗尽）

## 实验二：开启分页，建立页表

### 目标
建立初始页表，开启分页模式。此后 CPU 所有内存访问都经过地址翻译（虚拟地址 → 物理地址）。

### 背景知识：x86 分页机制

x86 保护模式的两级页表（32位，4KB页）：

```
虚拟地址（32位）：
  位31-22：页目录索引（10位，1024个目录项）
  位21-12：页表索引（10位，每个目录对应1024个页表项）
  位11-0：页内偏移（12位，4096字节）

地址翻译：
  CR3寄存器 → 页目录物理地址
  页目录[目录索引] → 页表物理地址
  页表[页表索引] → 物理页帧地址
  物理地址 = 页帧地址 + 偏移
```

**最简单的初始映射（恒等映射）：**

虚拟地址 = 物理地址（identity mapping）。内核启动时先用恒等映射，保证切换分页后代码还能继续执行（否则下一条指令的虚拟地址找不到对应物理地址，立刻崩溃）。

### 步骤

**1. 新建 `memory/vmm.h`**

```c
#ifndef VMM_H
#define VMM_H

#include <stdint.h>

#define PAGE_PRESENT  (1 << 0)   // 页存在
#define PAGE_WRITE    (1 << 1)   // 可写
#define PAGE_USER     (1 << 2)   // 用户态可访问（不设置则仅内核可访问）

typedef uint32_t page_entry_t;   // 页目录/页表项（4字节）

void vmm_init();
void vmm_map(uint32_t virt, uint32_t phys, uint32_t flags);  // 映射一个页
void vmm_unmap(uint32_t virt);                                // 取消映射

#endif
```

**2. 新建 `memory/vmm.c`**

内部数据：
```c
// 页目录（1024个条目，每条目4字节 = 4KB）
static page_entry_t page_directory[1024] __attribute__((aligned(4096)));

// 用于恒等映射的第一个页表（映射前4MB）
static page_entry_t first_page_table[1024] __attribute__((aligned(4096)));
```

实现 `vmm_init`：
```
// 清零页目录
memset(page_directory, 0, sizeof(page_directory));

// 建立恒等映射：虚拟地址 0~4MB → 物理地址 0~4MB
for i in 0..1024:
    first_page_table[i] = (i * PAGE_SIZE) | PAGE_PRESENT | PAGE_WRITE

// 把第一个页表注册到页目录的第0项
page_directory[0] = (uint32_t)first_page_table | PAGE_PRESENT | PAGE_WRITE

// 开启分页：把页目录地址写入 CR3，再设置 CR0 的 PG 位
asm volatile(
    "mov %0, %%cr3\n"          // 加载页目录地址
    "mov %%cr0, %%eax\n"
    "or $0x80000000, %%eax\n"  // 设置 PG 位（第31位）
    "mov %%eax, %%cr0\n"       // 开启分页
    : : "r"(page_directory) : "eax"
);
```

**3. 在 `kernel_main` 里开启分页**

```c
pmm_init();
vmm_init();        // 开启分页
vga_puts("Paging enabled.\n");
```

**注意：** 开启分页后，`vga_puts` 还能正常工作，证明恒等映射生效（VGA 显存在物理 0xB8000，恒等映射下虚拟地址也是 0xB8000）。

### 检查点
- [ ] 开启分页后内核不崩溃
- [ ] VGA 打印依然正常（证明恒等映射有效）
- [ ] 访问一个未映射的虚拟地址，触发 14 号中断（缺页异常）并打印错误信息（而不是三重错误崩溃）

## 实验三：实现内核堆，支持 kmalloc/kfree

### 目标
在物理内存管理和分页的基础上，实现堆分配器，让内核代码能用 `kmalloc(size)` 动态申请内存。

### 背景知识：堆分配器

MiniOS 里你用 `std::vector`、`std::list`，底层都是 `new`/`delete`，最终调用 `malloc`/`free`。裸机内核里没有这些，需要自己实现。

**最简单的堆：线性分配器（Bump Allocator）**

维护一个指针，每次分配就往后移动：

```
堆起始          当前位置
   ↓              ↓
   [已分配][已分配][------空闲------]
   
kmalloc(100): 返回当前位置，指针 +100
kmalloc(200): 返回当前位置，指针 +200
```

**缺点：** 无法释放（指针不能回退，否则会产生碎片）。

**更完整的堆：空闲链表（Free List）**

这就是 MiniOS `memory_manager` 的思想，只是用在内核堆上：

```c
struct heap_block {
    uint32_t     size;
    uint8_t      is_free;
    heap_block*  next;
};
```

### 步骤

**1. 新建 `memory/heap.h`**

```c
#ifndef HEAP_H
#define HEAP_H

#include <stdint.h>

void  heap_init(uint32_t start, uint32_t size);  // 初始化堆（起始地址，大小）
void* kmalloc(uint32_t size);                     // 分配内存
void  kfree(void* ptr);                           // 释放内存

#endif
```

**2. 新建 `memory/heap.c`，实现 First-Fit 分配（从 MiniOS 移植）**

内部结构：
```c
struct heap_block {
    uint32_t      magic;     // 魔数，用于检测堆损坏（如 0xDEADBEEF）
    uint32_t      size;      // 块大小（不含头部）
    uint8_t       is_free;
    struct heap_block* next;
};
```

`heap_init`：
```
创建一个大的空闲块，覆盖整个堆区域
block->magic = 0xDEADBEEF
block->size  = total_size - sizeof(heap_block)
block->is_free = 1
block->next  = NULL
```

`kmalloc(size)`：
```
遍历链表，找第一个 is_free=1 且 size >= 请求大小 的块

找到后：
  如果剩余空间 > sizeof(heap_block) + 最小块大小：
      切割：把剩余部分变成新的空闲块，插入链表
  标记当前块为已用（is_free = 0）
  返回块头部之后的地址（用户拿到的是数据区，不是头部）
```

`kfree(ptr)`：
```
ptr 往前移 sizeof(heap_block) 得到头部指针
检查 magic == 0xDEADBEEF（防止非法释放）
标记 is_free = 1
合并相邻空闲块（和 MiniOS 的 mergeAdjacentFree 完全一样）
```

**3. 确定堆的位置**

堆放在内核代码结束之后，链接脚本里加：

```ld
/* linker.ld 追加 */
PROVIDE(kernel_end = .);   /* 暴露内核结束地址给 C 代码 */
```

在 `kernel.c` 里：

```c
extern uint32_t kernel_end;       // 链接器提供的符号

#define HEAP_START ((uint32_t)&kernel_end)
#define HEAP_SIZE  (1024 * 1024)  // 1MB 内核堆

void kernel_main() {
    // ...
    vmm_init();
    heap_init(HEAP_START, HEAP_SIZE);
    
    // 测试
    void* p1 = kmalloc(100);
    void* p2 = kmalloc(200);
    kfree(p1);
    void* p3 = kmalloc(50);   // 应该复用 p1 释放的空间
    // 打印地址，验证 p3 == p1（首次适应，小于原块大小，重用）
}
```

### 检查点
- [ ] `kmalloc(100)` 返回非 NULL 地址
- [ ] 连续分配后地址连续（相差约 100 + 头部大小）
- [ ] `kfree` 后再分配，能重用已释放的内存
- [ ] 分配超出堆大小时返回 NULL（而不是崩溃）
- [ ] `magic` 检查能发现 double-free 或野指针释放

## 阶段总结

完成本阶段后，TinyOS 拥有了完整的内存管理体系：

| 层次 | 文件 | 功能 |
|------|------|------|
| 物理层 | `memory/pmm.h/c` | 位图管理所有物理页帧 |
| 虚拟层 | `memory/vmm.h/c` | 页表，虚拟地址→物理地址，开启分页 |
| 分配层 | `memory/heap.h/c` | kmalloc/kfree，内核动态内存 |

**与 MiniOS 的对应：**

| MiniOS | TinyOS |
|--------|--------|
| `MemoryManager::firstFit` | `heap.c::kmalloc`（算法完全一样） |
| `MemoryManager::free + mergeAdjacentFree` | `heap.c::kfree`（算法完全一样） |
| `new / std::vector` | `kmalloc`（裸机版） |
| **无对应** | `pmm.c`（物理页帧管理） |
| **无对应** | `vmm.c`（页表，开启分页） |

## 常见问题

**Q：开启分页后立刻三重错误？**  
A：最常见原因：页目录没有对齐到 4KB 边界（`__attribute__((aligned(4096)))` 是否加了？），或者 CR3 写入的地址不是页目录的物理地址。开启分页后的第一条指令必须在已映射的虚拟地址范围内。

**Q：`kmalloc` 应该分配多大的最小块？**  
A：至少要能容纳 `heap_block` 头部（约 16 字节），否则切割后剩余空间放不下头部，逻辑会出错。通常设置最小分配单元为 16 或 32 字节。

**Q：magic 值 0xDEADBEEF 有什么特殊含义吗？**  
A：没有，只是程序员惯用的魔数（容易辨认，不太可能随机出现）。也有用 `0xCAFEBABE`、`0xBAADF00D` 等。检查 magic 可以在调试时快速发现堆损坏（如写越界破坏了相邻块的头部）。
