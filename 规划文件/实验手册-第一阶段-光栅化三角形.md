> **阶段目标：** 编译通过，输出一张 TGA 图片，图片上有一个填充颜色的三角形。  
> **预计时间：** 2周，每天约1小时  
> **本阶段涉及文件：**
> ```
> game/
> ├── CMakeLists.txt          ← 本阶段创建
> └── src/
>     ├── main.cpp            ← 本阶段创建
>     ├── math/
>     │   └── vec.h           ← 本阶段创建（只写 Vec2、Vec3，Mat4 留给第二阶段）
>     └── core/
>         └── framebuffer.h   ← 本阶段创建
>         └── framebuffer.cpp ← 本阶段创建
> ```
> **本阶段不涉及：** `mat.h` / `texture` / `model` / `pipeline` / `shader`，这些留给后续阶段，现在不要创建。

## 实验一：搭建项目，让程序跑起来

### 目标
写一个最简单的 C++ 程序，用 CMake 编译，能运行并打印 "Hello, Renderer!"。

### 背景知识
CMake 是 C++ 项目最常用的构建工具。你不需要手动写编译命令，只需要写一个 `CMakeLists.txt` 描述"这个项目有哪些文件、用什么标准编译"，CMake 替你生成编译指令。

### 步骤

**1. 创建目录结构**

在 `game/` 下手动创建以下目录和空文件：
```
game/
├── CMakeLists.txt
└── src/
    ├── math/
    └── core/
    └── main.cpp
```

**2. 编写 `CMakeLists.txt`**

打开 `CMakeLists.txt`，你需要填写以下内容（每一行都理解它的含义再写）：

| 你需要声明的内容 | CMake 指令 | 说明 |
|---|---|---|
| CMake 最低版本 | `cmake_minimum_required(VERSION 3.10)` | 固定写法 |
| 项目名称 | `project(SoftRenderer)` | 随意起名 |
| C++ 标准 | `set(CMAKE_CXX_STANDARD 17)` | 使用 C++17 |
| 可执行文件由哪些源文件组成 | `add_executable(renderer src/main.cpp)` | 现在只有 main.cpp |
| 头文件搜索路径 | `target_include_directories(renderer PRIVATE src)` | 让编译器能找到 src/ 下的头文件 |

> ⚠️ **注意：** 后续每增加一个 `.cpp` 文件，都需要把它加进 `add_executable` 的列表里。这一点很重要，后续阶段会反复用到。

**3. 编写 `src/main.cpp`**

先写最简单的入口：
```cpp
#include <iostream>

int main() {
    std::cout << "Hello, Renderer!" << std::endl;
    return 0;
}
```

**4. 编译并运行**

在终端中执行（在 `game/` 目录下）：
```bash
mkdir build
cd build
cmake ..
make
./renderer
```

看到 `Hello, Renderer!` 输出，实验一完成。

### 检查点
- [ ] `cmake ..` 没有报错
- [ ] `make` 编译成功
- [ ] `./renderer` 输出了 `Hello, Renderer!`

## 实验二：理解 TGA 文件格式，输出纯色图片

### 目标
不借助任何图片库，用纯 C++ 写文件，输出一张全红色的 TGA 图片。

### 背景知识：TGA 文件格式

TGA 是一种极其简单的图片格式，适合我们手写。它的结构是：

```
[ 文件头：18个字节，固定格式，描述图片宽高、颜色位数等 ]
[ 像素数据：从左下角开始，每个像素3个字节（B、G、R顺序）]
```

文件头的18个字节含义如下：

| 字节位置 | 含义 | 本项目填什么值 |
|---|---|---|
| 0 | ID 字段长度 | 0 |
| 1 | 颜色表类型 | 0（不用颜色表） |
| 2 | 图像类型 | 2（未压缩真彩色） |
| 3-7 | 颜色表信息 | 全填 0 |
| 8-9 | X 起始坐标 | 0 |
| 10-11 | Y 起始坐标 | 0 |
| 12-13 | 图像宽度（像素） | 你的宽度，注意小端序 |
| 14-15 | 图像高度（像素） | 你的高度，注意小端序 |
| 16 | 每像素位数 | 24（RGB各8位） |
| 17 | 图像描述符 | 0 |

> **小端序说明：** 宽度如果是 800，用两个字节存储是 `0x20, 0x03`（低字节在前）。在 C++ 中，`short` 类型天然是小端序存储，直接写入即可。

### 步骤

**1. 编写 `src/core/framebuffer.h`**

定义 `Color` 结构体和 `Framebuffer` 类的接口：

```
需要定义：
- struct Color { uint8_t r, g, b; }
- class Framebuffer：
    - 构造函数：接受宽度和高度
    - void setPixel(int x, int y, Color color)  → 设置某个像素的颜色
    - void writeTGA(const std::string& filename)  → 把像素数据写成 TGA 文件
    - 私有成员：宽度、高度、存储所有像素的数组
```

思考：用什么数据结构存储所有像素？提示：`std::vector<Color>`，大小是 `width * height`。

**2. 编写 `src/core/framebuffer.cpp`**

实现上面声明的三个函数：

- **构造函数：** 初始化宽高，将像素数组初始化为全黑（r=0, g=0, b=0）。

- **setPixel：** 注意边界检查（x、y不能超出范围），然后将颜色写入数组。  
  像素数组是一维的，坐标 `(x, y)` 对应的下标是：`y * width + x`

- **writeTGA：**  
  用 `std::ofstream` 以二进制模式打开文件（注意：`std::ios::binary`）。  
  先写18字节的文件头，再逐像素写入像素数据。  
  注意：TGA 像素数据是 **BGR** 顺序，不是 RGB。

**3. 更新 `CMakeLists.txt`**

把 `src/core/framebuffer.cpp` 加入 `add_executable` 的文件列表。

**4. 修改 `src/main.cpp`**

```cpp
#include "core/framebuffer.h"

int main() {
    Framebuffer fb(512, 512);

    // 把所有像素设置为红色
    for (int y = 0; y < 512; y++) {
        for (int x = 0; x < 512; x++) {
            fb.setPixel(x, y, {255, 0, 0});
        }
    }

    fb.writeTGA("output.tga");
    return 0;
}
```

**5. 编译运行，用图片查看器打开 `output.tga`**

Mac 上双击 TGA 文件即可用预览（Preview）打开。

### 检查点
- [ ] 编译成功
- [ ] 生成了 `output.tga`
- [ ] 打开图片是一张 512×512 的纯红色图片

## 实验三：实现 Vec2、Vec3，画一条线

### 目标
实现数学向量类，然后用最简单的方式在图片上画一条直线。

### 背景知识
后续所有的坐标、方向、颜色计算都会用到向量。第一阶段只需要 Vec2（2D坐标）和 Vec3（3D坐标，同时也用来表示颜色RGB）。

**Mat4（4×4矩阵）暂时不要实现**，它属于第二阶段的内容。

### 步骤

**1. 编写 `src/math/vec.h`**

用 C++ 模板或者直接写两个结构体都可以。需要实现：

```
struct Vec2 {
    float x, y;
    需要：+ - * 运算符，点积，长度
};

struct Vec3 {
    float x, y, z;
    需要：+ - * 运算符，点积，叉积，长度，归一化
};
```

重点：**叉积（cross product）** 一定要实现，光栅化的重心坐标计算要用到它。

叉积公式：
```
a × b = (a.y*b.z - a.z*b.y,
         a.z*b.x - a.x*b.z,
         a.x*b.y - a.y*b.x)
```

对于二维向量，叉积退化为一个标量：`a.x*b.y - a.y*b.x`，这个值的符号可以判断两个向量的相对位置（顺时针/逆时针），光栅化会用到。

**2. 在 `main.cpp` 中用 `setPixel` 画一条线**

画线不需要 Bresenham（你之前做过），这里用最朴素的方法：
```
从点 A 到点 B，将 t 从 0.0 到 1.0 步进（步长 0.01）
每一步计算当前点坐标 = A + t*(B-A)
调用 setPixel 画出这个点
```

这个方法会有间隔（步长太大）或者性能差（步长太小），但**第一阶段不需要完美的画线算法**，能看到一条线就够了。

### 检查点
- [ ] `Vec2` 和 `Vec3` 编译通过
- [ ] 图片上出现一条白色的线

## 实验四：重心坐标法，填充三角形

### 目标
给定三角形的三个顶点，将三角形内部的所有像素填充为指定颜色。这就是**光栅化的核心**。

### 背景知识：重心坐标（Barycentric Coordinates）

**为什么不用扫描线法？**  
扫描线法也可以填充三角形，但重心坐标法在后续阶段有更大的用途：插值（颜色、UV坐标、深度值）都依赖重心坐标，所以必须用这个方法。

**什么是重心坐标？**

平面上一个点 P，相对于三角形 ABC，可以用三个权重 (α, β, γ) 表示：

```
P = α*A + β*B + γ*C
其中 α + β + γ = 1
```

如果 α、β、γ **都大于等于 0**，则点 P 在三角形内部（或边上）。

**如何计算重心坐标？**

利用面积法（或叉积法）：

```
三角形 ABC 的有向面积 = (B-A) × (C-A)  （这里是二维叉积，结果是标量）

α = 三角形 PBC 的有向面积 / 三角形 ABC 的有向面积
β = 三角形 APC 的有向面积 / 三角形 ABC 的有向面积
γ = 1 - α - β
```

二维叉积：对于向量 u=(ux,uy) 和 v=(vx,vy)，结果是 `ux*vy - uy*vx`。

### 步骤

**1. 在 `framebuffer.h/cpp` 中新增 `drawTriangle` 函数**

函数签名：
```cpp
void drawTriangle(Vec2 v0, Vec2 v1, Vec2 v2, Color color);
```

算法流程：

```
1. 计算三角形的包围盒（bounding box）
   xmin = min(v0.x, v1.x, v2.x)，xmax、ymin、ymax 同理
   （别忘了把包围盒裁剪到屏幕范围内）

2. 遍历包围盒内的每一个像素 (x, y)

3. 对每个像素，计算它相对于三角形的重心坐标 (α, β, γ)

4. 如果 α >= 0 && β >= 0 && γ >= 0，则调用 setPixel(x, y, color)
```

> **为什么要用包围盒？**  
> 遍历整个屏幕太慢了。包围盒把需要检测的像素范围限制在三角形周围的矩形区域内，这是光栅化的标准优化手段。

**2. 修改 `main.cpp` 测试**

```cpp
// 定义三个顶点（屏幕坐标，原点在左下角）
Vec2 v0 = {100, 100};
Vec2 v1 = {400, 100};
Vec2 v2 = {250, 400};

fb.drawTriangle(v0, v1, v2, {0, 255, 0}); // 绿色三角形
```

**3. 测试边界情况**

- 画两个相邻三角形，看看边缘是否有缝隙或重叠（正常情况下应该无缝拼接）
- 画一个很小的三角形（3个像素大小左右），看看还能不能填充

### 检查点
- [ ] 图片上出现一个填充颜色的三角形
- [ ] 三角形边缘没有明显的锯齿跳跃（像素级别的锯齿正常，不是bug）

## 常见问题

**Q：TGA 文件打不开？**  
A：检查文件头是否以二进制模式写入（`std::ios::binary`），文件头18字节是否完整，像素数据是否是 BGR 顺序。

**Q：三角形填充有空洞？**  
A：检查重心坐标的判断条件，确保用的是 `>= 0` 而不是 `> 0`（边缘像素也要填充）。

**Q：画出来的三角形上下颠倒？**  
A：TGA 默认原点在左下角，屏幕坐标原点通常在左上角。可以在 `writeTGA` 时翻转 y 轴，也可以在所有坐标计算中统一用左下角为原点。选一种方式，全程保持一致，不要混用。这个问题在后续阶段会一直存在，现在就要确定你的坐标系方向。
